## 1. 语义切块原理

### 1.1 为什么 RAG 一定要切块

原始文档通常不能直接整篇拿去检索或拼 prompt，原因有 3 个：

1. 检索粒度太粗
2. 上下文噪声太大
3. prompt 成本太高

所以 RAG 的第一原则是：

> 先把“长文档”变成“可检索的最小知识单元”。

### 1.2 切块本质上在解决什么问题

切块并不只是技术步骤，它其实在同时平衡 3 件事：

1. 检索精度
2. 上下文完整性
3. token 成本

如果 chunk 太大：

1. 召回会带入很多无关内容
2. 一个 chunk 里混了多个主题
3. rerank 和 generation 都更难判断证据边界

如果 chunk 太小：

1. 一个步骤可能被拆碎
2. 回答时上下文不完整
3. 同一个答案需要拼很多碎片

所以 chunk 设计从来不是“越小越好”或“越大越好”，而是：

> 让召回粒度和生成粒度尽量同时合理。

### 1.3 当前项目的切块策略

对应实现：

- [core/ingestion/chunkers/semantic_chunker.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/ingestion/chunkers/semantic_chunker.py)

当前项目不是简单固定长度切块，而是：

```text
section
  -> parent chunk
  -> child chunk
```

更具体地说：

1. 先按标题切 section
2. section 内优先按段落组织
3. 先切出较大的 parent chunk
4. 再把 parent 细切成 child chunk

### 1.4 为什么要 `parent + child` 两层

这是当前项目最关键的工程设计之一。

#### child chunk 的职责

1. 更短
2. 主题更聚焦
3. 更适合召回

#### parent chunk 的职责

1. 更完整
2. 更适合 rerank
3. 更适合 generation

为什么不只保留一种：

1. 只有大块，召回不够精细
2. 只有小块，生成时上下文容易碎

所以项目采用的是一个很经典的折中：

```text
child 负责“找到”
parent 负责“说明白”
```

### 1.5 overlap 的原理

项目里有两个 overlap 概念：

1. parent overlap
2. child overlap

它们共同解决的问题是：

> 句子、步骤、条件判断很容易刚好被切在边界。

如果完全不重叠，会发生：

1. 句子前半段在 chunk A
2. 后半段在 chunk B
3. 检索命中其中一个时，证据可能不完整

所以 overlap 的本质是：

> 用少量重复上下文，换取更稳定的语义连续性。

### 1.6 stable chunk_id 的原理

源码思路：

```python
sha256(prefix + doc_id + idx + content_prefix)
```

它不是随便生成一个 UUID，而是尽量让同一文档重复入库时，相同内容得到尽量稳定的 chunk_id。

这样做的价值：

1. 增量更新更稳
2. 引用更稳
3. 测试断言更稳
4. badcase 对比更稳

### 1.7 当前方案和其他切块方案怎么比较

#### 方案 A：固定长度硬切

优点：

1. 简单
2. 速度快

缺点：

1. 容易切断语义边界
2. 标题和正文容易分离
3. 引用可解释性差

#### 方案 B：当前项目的 section + parent/child

优点：

1. 更接近文档结构
2. 检索和生成分工更清楚
3. 对教学类、SOP 类文档更友好

缺点：

1. 实现更复杂
2. 对原文结构有一定依赖

#### 方案 C：纯语义模型切块

优点：

1. 可能更贴近语义边界

缺点：

1. 成本更高
2. 更难调试
3. 结果稳定性可能不如规则切块

### 1.8 如果以后升级，还可以怎么做

1. 基于 token 数而不是字符数切块
2. 对表格、代码块、FAQ 做结构化切块
3. 引入语义边界检测模型
4. 按文档类型动态选择切块策略

## 11. 企业安全检索与数据分级原理

### 11.1 为什么企业 RAG 不能只做“召回相关内容”

普通 demo 往往只回答一个问题：

> 哪些 chunk 和 query 最相关？

但企业场景必须同时回答第二个问题：

> 哪些 chunk 虽然相关，但当前用户根本不应该看到？

所以企业 RAG 的检索目标不是单一的相关性最大化，而是：

```text
在可访问范围内，找最相关的证据
```

这个区别非常关键。

如果你只做相关性，不做权限：

1. 召回质量可能看起来很好
2. 但系统会把“本不该给当前用户看的内容”也拿去生成
3. 这在企业环境里通常是不可接受的

### 11.2 当前项目为什么把 ACL 放在 retrieval 阶段，而不是 generation 之后

当前项目的设计原则已经明确成：

> 权限控制必须在检索前或检索中完成，不能等生成完再裁剪。

为什么：

1. 生成后裁剪并不能阻止敏感内容已经进入模型上下文
2. 一旦敏感上下文进了 prompt，就已经发生“出域”或“越权消费”
3. 企业系统需要的是“不能看到”，不是“看到后再遮掉”

所以当前实现采用的是两层防线：

```text
检索器 filter 下传
  + 本地 ACL fallback filter
```

### 11.3 access filter 和 structured filter 的分工

当前 retrieval 里有两类过滤条件：

#### A. structured filter

来源：

- `analyze_query`
- `resolve_context`
- `rewrite_query / build_query_plan`

作用：

- 缩小语义检索的业务范围
- 让“部门、时间、班次、版本”类问题召回更准

#### B. access filter

来源：

- `ChatRequest` 中的企业用户上下文
- `build_access_filters`
- `build_retrieval_acl_filters`

作用：

- 缩小当前用户可见范围
- 把 ACL 语义尽可能前置到 sparse / dense 检索阶段

你可以把两者记成：

```text
structured filter = 为了更准
access filter = 为了更安全
```

### 11.4 为什么当前项目要做 ACL fallback filter

底层检索器已经支持 `filters=...`，为什么还要在上层再做一次 `is_chunk_accessible()`？

原因在于：

1. 底层 filter 更像“候选缩小器”，不是强安全引擎
2. 不同 backend 对 metadata 的处理能力并不完全一致
3. 未来 Milvus、file backend、本地镜像之间可能存在 schema 和兼容差异

所以当前项目做的是：

```text
第一次过滤：尽量在检索器里少召回无权内容
第二次过滤：在 Python 层严格判断是否真的可访问
```

这就是为什么当前 `retrieve_docs_node` 在 child hits 阶段还要再跑一遍 ACL。

### 11.5 为什么 ACL fallback 放在 child hits，而不是 parent hits

当前项目采用 `child recall -> parent expand` 的检索结构。

因此 ACL 兜底放在 child hits 更合理，原因有 3 个：

1. child 是实际参与召回的最小搜索单元
2. child metadata 往往比 parent 更贴近原始索引信息
3. 如果等扩到 parent 再过滤，可能会把多个 child 的不同 ACL 语义混在一起

所以当前顺序是：

```text
召回 child
-> merge child
-> ACL fallback
-> expand to parent
-> fusion
```

这个顺序既兼顾安全，也兼顾当前 parent-child 设计。

### 11.6 数据分级为什么不是“命中文档里随便取一个”

当前 `data_classification` 的计算原则是：

```text
public < internal < sensitive < restricted
```

并且采用“命中结果取最高等级”的策略。

为什么不是取最低或平均：

1. 只要候选上下文里有一段 `restricted`，整次回答就应该按高等级处理
2. 企业安全里更常见的是“上限约束”，不是平均风险
3. 这有利于后续做 `model_route`、审计和前端提示

举个例子：

如果 5 个命中 chunk 里：

- 4 个是 `internal`
- 1 个是 `restricted`

那最终 `data_classification` 仍然应该是：

```text
restricted
```

### 11.7 model route 为什么先做“决策标签”，而不是直接切多模型

当前项目里的 `model_route` 还不是“真实切换了某个 LLM 执行器”，而是：

> 系统对这次回答应走什么安全策略的决策标签。

当前标签语义：

1. `external_allowed`
   允许走外部模型
2. `local_preferred`
   更倾向本地模型，但不一定已经接好
3. `local_only`
   只能走本地模型或拒答
4. `default`
   系统未启用模型路由时的默认值

为什么先做这个层次：

1. 先把系统状态、测试、API、日志、前端语义打通
2. 后面真接本地模型时，不必再返工 schema 和链路
3. 这符合工程上“先固化语义，再切换执行器”的稳妥做法

### 11.8 为什么最近两轮改造要临时绕过 fast path

当前 fast path 还主要服务于：

- Redis 热点答案缓存
- MySQL FAQ

它们当前的优点是：

1. 快
2. 稳
3. 成本低

但它们的不足也很明确：

1. 还没有真正接入 ACL
2. 还没有 data classification 语义
3. 还没有企业安全审计语义

所以当前项目选择：

```text
对带企业安全上下文的请求，先绕过 fast path
```

这不是倒退，而是一种典型的生产级保守策略：

> 在优化性能之前，先确保安全边界是正确的。

### 11.9 `access_denied` 和 `no_relevant_chunks` 为什么必须区分

这两个拒答原因在企业系统里语义完全不同。

#### `no_relevant_chunks`

表示：

- 系统在当前知识库里没找到足够相关的证据

#### `access_denied`

表示：

- 系统其实召回到了候选
- 但这些候选在 ACL / 数据分级过滤后都不可访问

如果把这两者混在一起，后果很严重：

1. 用户体验会误导
2. 审计日志会失真
3. 运维和 badcase 分析无法判断是真没数据，还是权限策略拦掉了

所以当前项目把这两个分支分开，是一个非常关键的企业化设计点。

### 11.10 当前方案和未来升级方向

当前方案属于：

```text
metadata-based ACL
classification-aware retrieval
route-tag-based model policy
```

优点：

1. 改动小
2. 易测试
3. 易和现有架构集成
4. 适合第一阶段企业化改造

局限：

1. ACL 语义还依赖 `metadata.extra`
2. 还没有真正本地模型执行层
3. fast path 尚未 ACL 化
4. 还没有引入更强的组织级权限系统

后续升级方向：

1. 继续把 ingestion 阶段的企业 metadata schema 做全，并补旧索引迁移
2. 对 FAQ / cache 层补 ACL
3. 引入更正式的 RBAC / ABAC / policy engine
4. 把 `model_route` 从标签升级成真实模型分发执行

说明：

这一轮其实已经完成了第一步的“基础版本”：

1. `BasicMetadataExtractor` 开始统一补 `owner_department / doc_type / data_classification / effective_date / authority_level / source_system`
2. `SemanticChunker` 会把这些字段稳定透传到 `ChunkMetadata.extra`
3. `metadata_filters` 已能识别列表型 metadata，因此 `allowed_departments / allowed_roles / project_ids` 不再只能走字符串模糊匹配

但它还不是最终形态，因为：

1. 旧索引数据还没有统一迁移
2. Milvus 一级 schema 目前还没全部覆盖这些新字段
3. 版本冲突、权威级别优先级还没有真正进入排序和生成策略

这一点在当前轮已经开始补齐：

1. `rerank_docs_node` 会在 cross-encoder 之后叠加 `authority_level / effective_date / version` 的治理优先级
2. 最终上下文会返回 `conflict_detected / conflict_summary`
3. `generate_answer_node` 会把这条治理提示一并传给模型，避免模型把多版本证据直接糊成一句话

同时这一轮又往前走了一步：

1. `data_classification` 已经开始真实控制出域上下文，而不是只留在 response 字段里
2. `restricted` 当前会在没有本地模型执行器时直接拒答
3. `internal / sensitive` 会在进入外部模型前做不同强度的脱敏与上下文裁剪

这一轮又补了一个更适合企业落地的过渡方案：

1. `local_only` 现在优先走本地受限模式占位执行，而不是默认直接失败
2. 这让系统在还没接正式本地大模型之前，就能先把“高敏数据不出域”的主链路讲通
3. 同时审计层新增了 `security_alert`，让高风险与高敏请求不只是可追踪，还能被单独分流处理
4. 当前又补了统一风控接口层，让“高风险请求识别、分级出域控制、拒答决策”不再散落在多个模块里
5. 这轮继续把 metadata schema 做厚，并把更多企业字段提升为 Milvus 一级过滤字段，为下一步 retrieval 性能优化打基础

在 ingestion 这一侧，这轮也有一个很关键的升级：

1. 当前不再是“所有文件统一按同一组切块参数处理”
2. 而是升级成“统一 `SemanticChunker` + 文件类型 profile”
3. 同时 parser 也会先增强结构信号：
   - `DOCX` 保留 heading level、列表和基础表格结构
   - `PPTX` 保留 bullet level
   - `CSV` 为每行补更强的主键标题

这套设计的好处是：

1. 保持统一 chunker，维护成本低
2. 又能让 PDF / PPTX / CSV / TXT 这些不同格式在切块参数上更贴各自形态
3. 后续如果 retrieval 发现某类文档效果差，可以先调 parser 输出或 profile，而不是马上重写整套 chunking 系统

再往下一步，这轮已经把 retrieval 性能优化真正接进主链路：

1. `analyze_query_node` 不再只产出“要不要 rewrite / hyde”，还会产出 `query_scene / preferred_retriever / top_k_profile / metadata_intent`
2. `retrieve_docs_node` 会把 `metadata_intent + structured_filters + access_filters` 合并成最终 retrieval filters
3. sparse / dense 的 `top_k` 会按 `precise / balanced / broad` 动态缩放
4. fusion 后会先按 metadata 命中情况做轻量 `metadata boost`
5. `rerank_docs_node` 只精排裁剪后的候选，而不是默认把所有 `fused_hits` 都交给 cross-encoder

这几步的意义不是“再加一层规则”，而是：

1. 尽量让更像企业真实答案的候选更早排前
2. 少让无关候选进入 rerank
3. 少让无效上下文进入生成
4. 同时优化质量、延迟和成本

这一轮又往前补了两步：

1. route pruning
   - 对 `precise` 问题主动裁掉低价值 route
   - 对 `preferred_retriever=sparse` 的问题减少 dense 侧无效调用
2. context packing
   - 在生成前限制文档数、单文档 chunk 数和总字符数
   - 同文档重复 section 不再重复进入 prompt

这两步的本质是一样的：

> 尽量把“少而精”的证据送到更贵的阶段。

这一轮又补了一层对外解释性：

1. citation 会带 `matched_routes / retrieval_score / semantic_score / governance_rank_score`
2. 同时再收敛成一条短的 `selection_reason`

这让系统不仅能回答“引用了哪里”，还能解释：

> 为什么这条证据在最终答案里被优先采用。

这一轮继续把 query understanding 本身也纳入 explainability：

1. `analyze_query_node` 现在会返回 `analysis_confidence / analysis_source / analysis_reason`
2. `analysis_source=heuristic`
   表示规则层已经足够高置信，不再额外消耗 LLM
3. `analysis_source=llm_enhanced`
   表示问题表达偏长尾或隐式，交给查询理解模型补了一次结构化判断
4. `analysis_source=*_guardrail`
   表示最终仍然不够稳，系统主动回到保守 `hybrid + balanced + no_hyde`

这一步的价值不是“让系统看起来更聪明”，而是让 query route 的决策过程也可以被评测、被回放、被解释。

这一轮又继续把它从“可观测”推进到“可调参”：

1. `/eval` 现在会额外产出 `query_understanding_report`
2. 它会汇总：
   - 高频 `query_scene`
   - 高频 `guardrail` 场景
   - 高频 `llm_enhanced` 场景
3. 同时给出一组轻量调参建议

这意味着后续不是靠感觉改规则，而是可以先看：

> 是哪类问题一直依赖 LLM，哪类问题一直不够稳，哪类问题值得优先补规则词典。

这一轮继续把“补规则词典”这件事正式做成了配置层：

1. 默认词典文件位于 `data/config/query_understanding_vocab.json`
2. `analyze_query` 不再只依赖源码里的固定 regex 词面
3. cache key 已经把 `QUERY_UNDERSTANDING_VOCAB_PATH` 带进去，避免词典切换后沿用旧分析结果

这一步的意义非常工程化：

- 让 badcase 回流后的修改更像“调配置”
- 降低频繁改源码的成本
- 为后续新疆能源内部术语扩展留入口

当前默认词典已经补到一版新疆能源专属内容：

1. 业务域新增 `fuel_management / power_generation / emergency_response / contract_management`
2. 部门别名新增 `燃料管理部 / 发电运行部 / 生产技术部 / 招标采购部`
3. 场站别名新增 `一号输煤线 / 集控室 / 南露天煤场 / 翻车机卸煤区`
4. 系统别名新增 `燃料管控平台 / 生产运营看板 / 设备缺陷管理系统`

这一轮继续把词典层从“关键词表”推进成“企业实体归一层”：

1. `department_aliases`
   把内部简称统一成规范部门名
2. `site_aliases`
   把矿区、厂站、场地简称统一成规范场站名
3. `system_aliases`
   把平台简称统一成规范系统名

它的价值不是把 query understanding 做成 NER，而是先把企业里最常见、最值钱、最可维护的实体归一做起来。

这一轮继续往前推进了一步：

1. 这些实体归一结果不再只停留在 `metadata_intent`
2. retrieval 在 fusion 后会额外做一层 `enterprise entity boost`
3. 当前重点提权的实体组包括：
   - `department`
   - `site`
   - `system`
   - `business_domain`
   - `process_stage / equipment / project`
4. 这样“安环部 / 二矿 / 安生平台”这类内部表达被归一后，会更明确地影响最终证据排序，而不是只做过滤
5. 对 `department / owner_department`、`plant / applicable_site` 这类同义实体组，过滤语义也已经从隐式 AND 调整成组内 OR，避免一份文档只写规范字段之一时被错误过滤

## 2. BM25 原理

### 2.1 BM25 是做什么的

BM25 是经典的稀疏检索算法，用来回答一个非常直接的问题：

> 这个 chunk 在词面上和当前 query 有多相关？

它的优势不在深层语义，而在：

1. 精确词匹配
2. 稀有词区分
3. 低成本高稳定

### 2.2 为什么 RAG 里一定要保留 BM25

企业问答里有很多问题本质上依赖精确词匹配：

1. 错误码
2. 命令
3. 文件路径
4. 产品名
5. 术语缩写

这些内容的共同点是：

> 一个词写对和写错，意义差别很大。

而这正是 BM25 擅长的。

### 2.3 BM25 的核心打分思想

BM25 主要综合 3 个因素：

1. TF：词在当前文档里出现频率
2. IDF：词在整个语料里是否稀有
3. Length Normalization：文档长度校正

可以把它粗略理解成：

```text
score(doc, query)
= 词频贡献
+ 稀有词加权
- 过长文档惩罚
```

更直观一点：

1. 词在当前 chunk 里出现得越集中，越可能相关
2. 词在整个知识库里越稀有，越有区分度
3. 不能因为长文里重复了很多次某个词，就无限抬高分数

### 2.4 一个最容易记住的例子

假设 query 是：

```text
E-1001 Redis 连接
```

候选 chunk：

```text
A: 系统报错 E-1001 时，请检查 Redis 连接和密码配置。
B: Redis 是高性能缓存系统，常用于会话存储。
```

为什么 A 更容易排前：

1. 同时命中了 `E-1001` 和 `Redis`
2. `E-1001` 在全库里是高区分度词
3. 文本主题直接围绕故障排查

### 2.5 当前项目里的 BM25 实现

对应文件：

- [core/retrieval/sparse_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/sparse_retriever.py)

核心流程：

```python
tokenize(text)
  -> BM25Okapi(corpus_tokens)
  -> get_scores(query_tokens)
```

### 2.6 为什么项目自己写了 `tokenize`

因为企业知识文本不是纯英文，也不是标准中文分词场景，而是常常混合：

1. 中文句子
2. 英文术语
3. 数字
4. 错误码
5. 路径片段

所以项目里用正则做了一个“够用且稳定”的 tokenizer。

它的目标不是语言学最优，而是工程最稳。

### 2.7 BM25 的局限

BM25 不擅长：

1. 语义近义表达
2. 口语化问法
3. 改写表达

例如：

```text
用户问：密码忘了怎么办
文档写：账号密码重置流程
```

词面不完全一致时，BM25 可能召回不稳。

### 2.8 BM25 和其他方案怎么选

#### 只用 BM25

优点：

1. 简单
2. 快
3. 对错误码、路径很稳

缺点：

1. 语义泛化弱

#### Elasticsearch / OpenSearch BM25

优点：

1. 更成熟
2. 支持更丰富查询能力

缺点：

1. 工程复杂度更高

#### 当前项目选择

项目把 BM25 当作“词面锚点召回器”，而不是唯一召回器。  
这就是最稳的用法。

### 2.9 可升级方向

1. 引入同义词词典
2. 增加 query expansion
3. 迁移到 Elasticsearch / OpenSearch
4. 结合字段权重检索

## 3. Dense 检索原理

### 3.1 Dense 检索解决什么问题

Dense 检索解决的是：

> 词不一定一样，但意思接近。

它最适合的场景是：

1. 用户问法口语化
2. 文档写法更规范
3. query 和文档存在同义改写

### 3.2 核心思想

Dense 检索先把：

1. query
2. document chunk

都编码到同一个向量空间里，然后通过向量相似度判断它们是否靠近。

本质上，它不直接比“词”，而是在比“语义位置”。

### 3.3 为什么向量空间能表达语义相似

embedding 模型训练时，会尽量把：

1. 语义相近文本拉近
2. 语义不相关文本拉远

所以类似下面这种情况，Dense 往往比 BM25 更稳：

```text
query: 密码忘了怎么办
doc: 账号密码重置流程
```

### 3.4 当前项目的实现

对应文件：

- [core/retrieval/dense_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/dense_retriever.py)

关键代码：

```python
self._get_model().encode(texts, normalize_embeddings=True)
sims = self._matrix @ q
```

### 3.5 为什么 `normalize_embeddings=True` 很重要

余弦相似度公式：

```text
cos(a, b) = a · b / (|a||b|)
```

当向量都归一化后：

```text
|a| ≈ 1, |b| ≈ 1
=> cos(a, b) ≈ a · b
```

这时直接做矩阵乘法就可以近似计算余弦相似度。

好处：

1. 算法简单
2. 批量计算高效
3. 实现清晰

### 3.6 当前项目为什么也是只索引 child chunk

和 BM25 一样，Dense 也只把 `searchable=True` 的 chunk 放进检索矩阵。

原因：

1. child 更适合召回
2. parent 太长时语义中心容易被摊薄
3. 检索阶段需要“敏捷命中”，不是“完整解释”

### 3.7 Dense 的优势

1. 处理语义近义表达
2. 处理 paraphrase
3. 对自然语言问法更鲁棒

### 3.8 Dense 的局限

1. 对错误码、路径、命令等精确 token 未必更强
2. 分数可解释性不如 BM25
3. 模型切换后，阈值和效果可能整体漂移

### 3.9 Dense 和其他向量方案怎么选

#### 本地矩阵 + SentenceTransformer

优点：

1. 简单
2. 好调试
3. 适合学习和中小规模数据

缺点：

1. 不适合超大规模

#### 向量数据库 + ANN

优点：

1. 更适合大规模召回
2. 查询更快

缺点：

1. 工程复杂度更高
2. 学习成本更高

### 3.10 可升级方案

1. 使用更强 embedding 模型
2. 接入 ANN 索引
3. 引入多向量检索
4. 按 query 类型选不同 embedding 模型

## 4. 混合召回与融合原理

### 4.1 为什么要混合召回

因为：

1. BM25 看词面
2. Dense 看语义

两者各有盲区。

如果只信其中一个：

1. 词面问题可能漏掉
2. 语义近义问题也可能漏掉

所以混合召回的本质不是“多搞一层复杂度”，而是：

> 同时覆盖两类相关性。

### 4.2 当前项目的融合入口

对应文件：

- [core/retrieval/hybrid_fusion.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/hybrid_fusion.py)

### 4.3 RRF 原理

RRF 全称 Reciprocal Rank Fusion。

它不比较原始分数，而是比较排名。

基础公式可以理解成：

```text
RRF score = Σ 1 / (k + rank_i)
```

假设某个 chunk：

1. 在 BM25 里排第 1
2. 在 Dense 里排第 3

那么：

```text
score = 1/(60+1) + 1/(60+3)
```

### 4.4 为什么 RRF 这么好用

因为不同检索器的原始分数通常不可比。

例如：

1. BM25 分数不是概率
2. 向量相似度也不是概率
3. 两者量纲完全不同

RRF 直接绕开这个问题，只看排名。

这意味着：

> 只要一个 chunk 在多路召回里都排得不错，它就会自然被抬起来。

### 4.5 Weighted Fusion 原理

Weighted Fusion 的思路是：

1. 先对不同召回器分数做归一化
2. 再做加权和

示意：

```text
final = sparse_weight * sparse_score + dense_weight * dense_score
```

它的优点是：

1. 可控性更强
2. 便于表达业务偏好

它的缺点是：

1. 需要调权重
2. 归一化方式会影响结果
3. 阈值更难解释

### 4.6 当前项目默认为什么更适合用 RRF

因为当前项目更强调：

1. 稳妥默认值
2. 教学可理解性
3. 少调参也能跑

所以 RRF 更适合做默认基线。

### 4.7 混合融合之后为什么还要门控

因为“融合出了结果”不等于“结果足够可靠”。

门控的本质是：

> 判断这些候选是否值得继续投入 rerank 和 generation 成本。

如果候选明显空或很弱，最稳的做法不是继续生成，而是尽早拒答。

### 4.8 可升级方案

1. 基于 `strategy_signals` 动态切换融合策略
2. 加入学习排序
3. 引入文档新鲜度、来源可信度等特征
4. 扩展到多路召回融合

## 5. Cross-Encoder Rerank 原理

### 5.1 为什么召回后还要 rerank

召回阶段的目标是：

1. 别漏掉候选

重排阶段的目标是：

1. 从候选里挑出真正最值得给模型看的少量片段

所以二者目标不同：

```text
召回追求 recall
重排追求 precision
```

### 5.2 Cross-Encoder 的原理

Cross-Encoder 不像 Dense 那样分别编码 query 和 doc。  
它会把 query 和候选 chunk 一起送进模型。

也就是说，模型直接看的是：

```text
[query, chunk]
```

然后输出一个“这两者有多相关”的分数。

### 5.3 它为什么通常比简单相似度更准

因为它能建模更细粒度的交互关系：

1. query 里的某个词和 chunk 里的哪个句子对应
2. 两者是否真的在回答同一件事
3. 上下文里是否存在关键条件或否定关系

这也是为什么 Cross-Encoder 往往比 Dual-Encoder 更适合做精排。

### 5.4 当前实现

对应文件：

- [core/retrieval/reranker.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/reranker.py)

关键代码：

```python
pairs = [[query, c.content] for c in candidates]
scores = self._get_model().predict(pairs, show_progress_bar=False)
```

### 5.5 为什么它更准但更慢

因为它不是提前把所有文档向量算好再查，而是对每个候选都要做一次联合推理。

所以它适合：

1. 小规模候选精排

不适合：

1. 全库大规模召回

### 5.6 一个容易理解的例子

query：

```text
怎么处理 E-1001
```

两个候选：

```text
A: E-1001 常见原因与修复步骤
B: Redis 基础介绍与安装指南
```

召回阶段 A 和 B 都可能进来。  
但 Cross-Encoder 在联合看 `query + A`、`query + B` 后，通常会更稳地把 A 排前。

### 5.7 rerank 和 LLM 自己判断有什么区别

LLM 当然也能看上下文判断，但：

1. 直接把很多噪声上下文喂给 LLM 成本高
2. 也更容易被噪声诱导

rerank 的价值就在于：

> 在 generation 之前，先把候选压缩成更干净的证据集合。

### 5.8 可升级方案

1. 更轻量的 reranker
2. 分层 rerank
3. 服务化 rerank
4. 按 query 类型选择是否 rerank

## 6. Grounded Generation 原理

### 6.1 什么叫 grounded generation

Grounded generation 的意思是：

> 模型回答必须建立在当前提供的上下文之上，而不是凭常识自由发挥。

这也是 RAG 和普通聊天系统最核心的差别之一。

### 6.2 当前项目是怎么实现 grounded generation 的

对应文件：

- [core/generation/prompts/templates.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/generation/prompts/templates.py)
- [core/generation/context_format.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/generation/context_format.py)
- [core/orchestration/nodes/generate_answer.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/orchestration/nodes/generate_answer.py)

当前项目主要用 4 个手段约束模型：

1. 只喂重排后的上下文
2. prompt 明确要求只基于 CONTEXT 回答
3. 每段上下文都带 `CHUNK_ID`
4. 要求输出 `CITATIONS_JSON`

### 6.3 为什么 `CHUNK_ID` 很关键

因为 grounded 不只是“心理上相信模型有看上下文”，而是要能落到程序可校验。

`CHUNK_ID` 的价值：

1. 让模型引用时有稳定锚点
2. 后处理能做白名单校验
3. 前端能展示明确来源

### 6.4 为什么 `CITATIONS_JSON` 比纯文本引用更稳

如果只让模型输出：

```text
[CHUNK_ID:abc]
```

后处理只能靠正则去猜。  
而结构化 `CITATIONS_JSON` 更适合：

1. 稳定解析
2. 程序校验
3. 前端展示

### 6.5 当前方案的本质

本质上，当前项目是在把一个开放式生成问题收敛成：

```text
受限上下文
+ 可追溯引用
+ 结构化输出
```

这就是为什么它比“直接让模型回答”更接近生产级系统。

### 6.6 当前实现的局限

1. 仍然依赖模型遵守格式
2. 上下文过长时稳定性会下降
3. 目前引用粒度还是 chunk 级

### 6.7 可升级方案

1. JSON schema / function calling
2. 句子级引用
3. 答案-证据对齐评分
4. 更细的拒答模板

## 7. Grounding 校验原理

### 7.1 为什么模型输出后还要再校验

因为模型即使被 prompt 约束，仍然可能：

1. 漏引用
2. 引错引用
3. 编造引用
4. 低置信度胡说

所以 grounded generation 不等于 grounded answer。  
中间还必须有系统校验层。

### 7.2 当前项目的做法

对应文件：

- [core/orchestration/nodes/validate_grounding.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/orchestration/nodes/validate_grounding.py)

检查内容：

1. 非拒答答案是否带 citations
2. citations 是否属于当前上下文
3. confidence 是否低于系统阈值
4. 记录 citation coverage

### 7.3 这一层的工程意义

它把两件事彻底分开了：

1. 模型说了什么
2. 系统是否认可它

这就是企业系统和简单 Demo 的关键差别。

### 7.4 三种典型失败场景

#### 场景 A：没有引用

说明模型没有给出证据。  
系统通常应该下调置信度，必要时直接拒答。

#### 场景 B：引用不在当前上下文里

说明模型可能在编造引用。  
这种情况比“答错”更危险，因为它看起来像有证据。

#### 场景 C：置信度低于阈值

即使文本通顺，系统也可能拒答。  
这是在控制风险，而不是在追求“每次都要回答”。

### 7.5 为什么这是 RAG 的关键防线

如果没有这一层：

1. 低质量证据也能输出漂亮答案
2. 编造引用不容易被发现
3. 前端看到的会是“看起来像真的”结果

所以 grounding 校验不是装饰，而是：

> 模型输出进入用户界面之前的最后一道闸门。

### 7.6 可升级方案

1. 更细粒度证据校验
2. 句子级 grounding
3. 事实核验模型
4. 引入 answer-evidence alignment scorer

## 8. RAGAS 评测原理

### 8.1 为什么需要 RAGAS

因为 RAG 的质量不是一个单一维度问题。  
你至少要同时关心：

1. 答案是否忠实于证据
2. 答案是否答到点上
3. 是否召回了关键证据
4. 召回里是否混入太多噪声

### 8.2 当前项目的评测流程

对应文件：

- [core/evaluation/ragas_runner.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/evaluation/ragas_runner.py)

流程：

1. 读 JSONL 评测集
2. 对每个问题跑当前 RAG 系统
3. 收集 `answer` 和 `contexts`
4. 构造 HuggingFace `Dataset`
5. 调用 `ragas.evaluate`

### 8.3 当前使用的 4 个核心指标

#### Faithfulness

答案是否忠于上下文。

如果这个分数低，优先怀疑：

1. 幻觉
2. prompt 约束不够
3. grounding 失效

#### Answer Relevancy

答案是否真正回答了问题。

如果这个分数低，优先怀疑：

1. 回答跑偏
2. query 理解有问题
3. 生成组织方式不佳

#### Context Recall

系统是否召回了支持答案所需的关键证据。

如果这个分数低，优先怀疑：

1. 切块
2. query rewrite
3. 检索
4. fusion / rerank

#### Context Precision

召回上下文是否足够干净，而不是噪声很多。

如果这个分数低，优先怀疑：

1. top_k 太大
2. 检索太宽
3. rerank 不够准
4. chunk 太大

### 8.4 为什么不能只看 faithfulness

因为：

1. 一个答案可能很忠实，但没有真正回答问题
2. 一个答案也可能看起来相关，但证据召回其实不足

所以 RAG 评测应该永远是多指标联合看。

### 8.5 指标组合怎么帮助定位根因

#### `context_recall` 低 + `context_precision` 低

优先怀疑检索整体有问题。

#### `context_recall` 高 + `faithfulness` 低

优先怀疑证据到了，但模型没老实基于证据回答。

#### `context_recall` 高 + `faithfulness` 高 + `answer_relevancy` 低

优先怀疑答案虽然 grounded，但没真正答到点上。

### 8.6 当前实现为什么先串行执行

当前项目在教学环境里优先保证：

1. 日志更清楚
2. 错误更容易定位
3. 单题 badcase 更容易回溯

所以先串行比一开始就上高并发更适合学习。

### 8.7 可升级方案

1. 增加业务级评测集
2. 建立版本回归对比
3. 增加 badcase 分类体系
4. 建立人工评测闭环

## 9. 可观测性原理

### 9.1 为什么 RAG 项目更需要可观测性

因为 RAG 不是单一模型调用，而是多阶段系统：

1. API
2. 检索
3. 融合
4. 重排
5. 生成
6. 后处理

任何一层都可能：

## 10. 查询规划与 Milvus 后端原理

这一章专门讲你刚才指出的那条链路：

```text
策略选择模型 -> 多路查询 -> 向量检索后端 -> rerank -> answer
```

如果只讲 BM25、Dense、Fusion，而不讲“查询规划”和“Dense backend 分发”，你就会看到很多技术点，但看不清它们在真实问答链路里是怎么接起来的。

### 10.1 为什么不能只有一条 query

很多 RAG 教学资料会把检索写成：

```text
question -> rewrite -> retrieve
```

但生产场景里，一个问题往往同时包含多个检索切面。

例如：

```text
我有一个包含 100 亿条记录的数据集，想把它存到 Milvus 中进行查询，可以吗？
```

这个问题里同时混着：

1. 产品名：`Milvus`
2. 规模约束：`100 亿条记录`
3. 任务目标：`存进去并查询`
4. 判断诉求：`可不可以`

如果只保留一条 query，很容易只覆盖其中一个切面。

### 10.2 当前项目里的“策略选择模型”到底是什么

当前项目没有把它实现成一个名字就叫“策略选择模型”的类。

它实际上是由 4 层协作组成的：

1. `analyze_query_node`
   抽取 `strategy_signals`，例如是否依赖历史、是否需要 keyword boost、是否像结构化事实查询
2. `clarify_query_node`
   判断当前问题是否缺少关键槽位，需不需要先追问
3. `resolve_context_node`
   在多轮场景下生成 `resolved_query`
4. `build_query_plan`
   基于 `question + resolved_query + strategy_signals` 生成多路查询计划

也就是说，当前项目把“策略选择”拆成：

```text
策略信号抽取
  + 缺槽位闸门
  + 上下文补全
  + 多路查询规划
```

这样做的好处是：

1. 每一步职责更清楚
2. 出问题时更容易排查
3. 更适合企业问答里的多轮、省略、时间/部门/班次/人员类问题

### 10.3 `build_query_plan` 为什么是关键技术点

对应文件：

- [core/orchestration/query_expansion.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/orchestration/query_expansion.py)

它输出的不是一个字符串，而是一份结构化检索计划：

```python
QueryPlan(
    resolved_query=...,
    rewritten_query=...,
    multi_queries=[...],
    keyword_queries=[...],
    hyde_query="...",
    structured_filters={...},
    planning_summary="...",
)
```

这些字段分别解决不同问题：

1. `resolved_query`
   解决多轮承接、省略和代词补全的问题
2. `rewritten_query`
   解决用户表达口语化、不适合检索的问题
3. `multi_queries`
   解决复杂问题只有一个检索视角不够的问题
4. `keyword_queries`
   解决编号、术语、设备名、字段名需要精确词面命中的问题
5. `hyde_query`
   解决 query 太短、太抽象时 Dense 语义锚点不够的问题
6. `structured_filters`
   解决时间、部门、班次、人员、环境、版本等约束需要进入 metadata filter 的问题
7. `planning_summary`
   解决调试时“这次为什么这样搜”的可解释性问题

### 10.4 多路查询各自适合什么技术

#### `resolved_query`

作用：

1. 先把多轮问题补全成完整句子
2. 通常同时喂给 sparse 和 dense

#### `rewritten_query`

作用：

1. 把口语化问题收敛成更正式、更可检索的表达
2. 通常同时喂给 sparse 和 dense

#### `multi_queries`

作用：

1. 把一个复杂问题拆成多个检索切面
2. 扩大召回覆盖面

#### `keyword_queries`

作用：

1. 把高区分度 token 单独拿出来
2. 提高 BM25 的精确命中能力

适合场景：

1. 错误码
2. 路径
3. 产品名
4. 配置项
5. 缩写词

#### `hyde_query`

作用：

1. 先构造一个“可能的答案描述”
2. 再让 Dense 用这个描述去找语义接近的 chunk

#### `structured_filters`

作用：

1. 把时间、部门、班次、人员、环境、版本这类约束变成 metadata filter
2. 让 sparse / dense / Milvus dense 都可以在召回时先缩小候选范围
3. 对于 Milvus，常见企业字段现在会优先转成 collection 一级字段过滤表达式，而不是只做召回后本地筛

### 10.5 HyDE 的原理到底是什么

HyDE 可以理解成：

```text
先写一段“可能长什么样的相关文档描述”，再把这段描述拿去做 embedding 检索
```

为什么这样可能有效：

1. 原问题有时太短
2. 原问题只有需求，没有知识表达
3. 文档库里的真正相关内容往往是陈述句、步骤句、说明句

举例：

```text
原问题：这个错误怎么处理
HyDE query：错误码排查通常需要先确认报错编号、环境、日志，并检查相关依赖服务状态
```

这样 Dense 更容易在语义空间里找到“故障排查类 chunk”。

它的风险也很明显：

1. 假设答案可能带偏
2. 如果模型幻觉很强，反而会污染召回方向

所以当前项目里 `hyde_query` 不是强制主路径，而是可选 route。

### 10.6 多路查询为什么最终要统一融合

因为不同 route 的结果分布不同：

1. `keyword_queries` 往往命中更精确，但覆盖面窄
2. `multi_queries` 往往覆盖面广，但噪声更多
3. `hyde_query` 可能召回语义相关但词面不一致的片段

如果直接“谁命中谁赢”，系统会非常不稳。

所以当前项目采用的是：

```text
route 内部先检索
  -> 所有 route 结果统一汇总
  -> child 粒度去重
  -> parent 回扩
  -> fusion
  -> rerank
```

本质上是在做：

> 先把不同路线都变成候选证据，再交给统一排序层决策。

### 10.7 为什么 Dense backend 要抽象成 file / milvus 两种实现

对应文件：

- [core/services/runtime.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/services/runtime.py)
- [core/retrieval/dense_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/dense_retriever.py)
- [core/retrieval/milvus_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/milvus_retriever.py)

核心装配逻辑：

```python
if self.settings.vector_backend == "milvus":
    self._dense = MilvusDenseRetriever(self.settings)
else:
    self._dense = DenseRetriever(self.settings)
```

这里的设计思想非常重要：

1. 对编排层暴露统一接口
2. 把 backend 选择放到 runtime 装配层
3. 上层节点不关心底层向量是本地矩阵还是远端向量库

这是一种很典型的工程解耦思路：

```text
编排层依赖能力接口
运行时决定具体实现
```

这次升级后，`MilvusDenseRetriever` 不再只是“远端向量搜索替身”，还承担了两件更贴近生产的事：

1. 把 `department / shift / line / person / time / environment / version / doc_category`
   这类企业字段提升为 collection 一级 schema 字段
2. 在运行时发现旧 collection 缺少这些字段时，基于本地快照自动重建远端 collection

### 10.8 为什么接了 Milvus 之后，本地 `IndexStore` 还要保留

当前项目不是：

```text
用了 Milvus 就把本地索引全部删掉
```

而是：

```text
本地 IndexStore 继续保留
  + Milvus 负责 dense retrieval
```

原因有 4 个：

1. 当前 parent 回扩仍然需要本地 chunk 可读镜像
2. parent 回扩仍然需要本地 chunk metadata
3. 调试时本地 `chunks.jsonl` 比远端 collection 更容易直接看
4. 服务重启时可以用本地快照补建 Milvus collection

这就是当前项目采用的工程折中：

```text
本地镜像负责“可读、可回查、可调试”
Milvus 负责“正式的 dense 向量召回”
BGEM3 优先负责“统一的 dense + sparse 检索表示”
```

### 10.8.1 为什么这一轮把检索编码优先切到 BGEM3

对应文件：

- [core/retrieval/bgem3_backend.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/bgem3_backend.py)
- [core/retrieval/dense_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/dense_retriever.py)
- [core/retrieval/sparse_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG- project/enterprise-rag-platform/core/retrieval/sparse_retriever.py)

这次切 BGEM3 的核心原因不是“跟风换模型”，而是要解决当前 hybrid retrieval 的一个长期问题：

```text
dense 表示来自 SentenceTransformer
sparse 表示来自 BM25
```

这两条路线都能工作，但它们天然来自两套不同的表示空间：

1. dense 更擅长语义近邻
2. BM25 更擅长词面强锚点
3. 两边的分布、偏好和边界并不一致

而 `BGEM3EmbeddingFunction` 的优势是：

1. 同一个编码器同时产出 dense 和 sparse 表示
2. 中文、短文本、术语和制度问句的一致性通常更好
3. 更适合后续往 Milvus 官方 hybrid search 路线继续演进

所以当前项目这一轮改成：

```text
BGEM3 可用：
  dense -> BGEM3 dense
  sparse -> BGEM3 sparse

BGEM3 不可用：
  dense -> SentenceTransformer
  sparse -> BM25
```

这样做的好处是：

1. 不会因为依赖缺失把现有主链路打挂
2. 代码层已经先切到了 BGEM3-first
3. 生产环境补齐 `pymilvus[model] + FlagEmbedding` 后，系统就能直接吃到 BGEM3 检索收益

### 10.9 如果做技术选型，这几种方案怎么比

#### 方案 A：只有本地 DenseRetriever

优点：

1. 简单
2. 本地调试方便
3. 学习成本低

缺点：

1. 数据规模上来后内存压力大
2. 检索能力和工程能力都有限

#### 方案 B：所有向量能力直接全切 Milvus

优点：

1. 更接近正式生产架构

缺点：

1. 调试成本更高
2. 需要额外解释 parent 回扩、BM25 语料来源、索引同步

#### 方案 C：当前项目方案，本地镜像 + 可切换 Dense backend

优点：

1. 学习友好
2. 工程过渡平滑
3. 对现有链路侵入小

缺点：

1. 需要维护两份语义上相关的数据视图
2. 需要关注本地快照和 Milvus collection 的一致性

### 10.10 以后还能怎么升级

1. 增加真正的 query router，为不同问题类型动态选择 route 集合
2. 让不同 route 使用不同 top_k、不同 fusion 权重
3. 对 `hyde_query` 做开关和质量监控，避免低质量 route 污染召回
4. 给 Milvus 增加 collection 分片、分租户或字段过滤能力
5. 把 route 级 trace 打到可观测性系统里，直接分析哪条 route 真正贡献了最终答案
