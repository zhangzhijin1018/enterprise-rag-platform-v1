## 1. 语义切块原理

### 1.1 为什么 RAG 一定要切块

原始文档通常不能直接整篇拿去检索或拼 prompt，原因有 3 个：

1. 检索粒度太粗
2. 上下文噪声太大
3. prompt 成本太高

所以 RAG 的第一原则是：

> 先把“长文档”变成“可检索的最小知识单元”。

### 1.2 切块本质上在解决什么问题

切块并不只是技术步骤，它其实在同时平衡 3 件事：

1. 检索精度
2. 上下文完整性
3. token 成本

如果 chunk 太大：

1. 召回会带入很多无关内容
2. 一个 chunk 里混了多个主题
3. rerank 和 generation 都更难判断证据边界

如果 chunk 太小：

1. 一个步骤可能被拆碎
2. 回答时上下文不完整
3. 同一个答案需要拼很多碎片

所以 chunk 设计从来不是“越小越好”或“越大越好”，而是：

> 让召回粒度和生成粒度尽量同时合理。

### 1.3 当前项目的切块策略

对应实现：

- [core/ingestion/chunkers/semantic_chunker.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/ingestion/chunkers/semantic_chunker.py)

当前项目不是简单固定长度切块，而是：

```text
section
  -> parent chunk
  -> child chunk
```

更具体地说：

1. 先按标题切 section
2. section 内优先按段落组织
3. 先切出较大的 parent chunk
4. 再把 parent 细切成 child chunk

### 1.4 为什么要 `parent + child` 两层

这是当前项目最关键的工程设计之一。

#### child chunk 的职责

1. 更短
2. 主题更聚焦
3. 更适合召回

#### parent chunk 的职责

1. 更完整
2. 更适合 rerank
3. 更适合 generation

为什么不只保留一种：

1. 只有大块，召回不够精细
2. 只有小块，生成时上下文容易碎

所以项目采用的是一个很经典的折中：

```text
child 负责“找到”
parent 负责“说明白”
```

### 1.5 overlap 的原理

项目里有两个 overlap 概念：

1. parent overlap
2. child overlap

它们共同解决的问题是：

> 句子、步骤、条件判断很容易刚好被切在边界。

如果完全不重叠，会发生：

1. 句子前半段在 chunk A
2. 后半段在 chunk B
3. 检索命中其中一个时，证据可能不完整

所以 overlap 的本质是：

> 用少量重复上下文，换取更稳定的语义连续性。

### 1.6 stable chunk_id 的原理

源码思路：

```python
sha256(prefix + doc_id + idx + content_prefix)
```

它不是随便生成一个 UUID，而是尽量让同一文档重复入库时，相同内容得到尽量稳定的 chunk_id。

这样做的价值：

1. 增量更新更稳
2. 引用更稳
3. 测试断言更稳
4. badcase 对比更稳

### 1.7 当前方案和其他切块方案怎么比较

#### 方案 A：固定长度硬切

优点：

1. 简单
2. 速度快

缺点：

1. 容易切断语义边界
2. 标题和正文容易分离
3. 引用可解释性差

#### 方案 B：当前项目的 section + parent/child

优点：

1. 更接近文档结构
2. 检索和生成分工更清楚
3. 对教学类、SOP 类文档更友好

缺点：

1. 实现更复杂
2. 对原文结构有一定依赖

#### 方案 C：纯语义模型切块

优点：

1. 可能更贴近语义边界

缺点：

1. 成本更高
2. 更难调试
3. 结果稳定性可能不如规则切块

### 1.8 如果以后升级，还可以怎么做

1. 基于 token 数而不是字符数切块
2. 对表格、代码块、FAQ 做结构化切块
3. 引入语义边界检测模型
4. 按文档类型动态选择切块策略

## 2. BM25 原理

### 2.1 BM25 是做什么的

BM25 是经典的稀疏检索算法，用来回答一个非常直接的问题：

> 这个 chunk 在词面上和当前 query 有多相关？

它的优势不在深层语义，而在：

1. 精确词匹配
2. 稀有词区分
3. 低成本高稳定

### 2.2 为什么 RAG 里一定要保留 BM25

企业问答里有很多问题本质上依赖精确词匹配：

1. 错误码
2. 命令
3. 文件路径
4. 产品名
5. 术语缩写

这些内容的共同点是：

> 一个词写对和写错，意义差别很大。

而这正是 BM25 擅长的。

### 2.3 BM25 的核心打分思想

BM25 主要综合 3 个因素：

1. TF：词在当前文档里出现频率
2. IDF：词在整个语料里是否稀有
3. Length Normalization：文档长度校正

可以把它粗略理解成：

```text
score(doc, query)
= 词频贡献
+ 稀有词加权
- 过长文档惩罚
```

更直观一点：

1. 词在当前 chunk 里出现得越集中，越可能相关
2. 词在整个知识库里越稀有，越有区分度
3. 不能因为长文里重复了很多次某个词，就无限抬高分数

### 2.4 一个最容易记住的例子

假设 query 是：

```text
E-1001 Redis 连接
```

候选 chunk：

```text
A: 系统报错 E-1001 时，请检查 Redis 连接和密码配置。
B: Redis 是高性能缓存系统，常用于会话存储。
```

为什么 A 更容易排前：

1. 同时命中了 `E-1001` 和 `Redis`
2. `E-1001` 在全库里是高区分度词
3. 文本主题直接围绕故障排查

### 2.5 当前项目里的 BM25 实现

对应文件：

- [core/retrieval/sparse_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/sparse_retriever.py)

核心流程：

```python
tokenize(text)
  -> BM25Okapi(corpus_tokens)
  -> get_scores(query_tokens)
```

### 2.6 为什么项目自己写了 `tokenize`

因为企业知识文本不是纯英文，也不是标准中文分词场景，而是常常混合：

1. 中文句子
2. 英文术语
3. 数字
4. 错误码
5. 路径片段

所以项目里用正则做了一个“够用且稳定”的 tokenizer。

它的目标不是语言学最优，而是工程最稳。

### 2.7 BM25 的局限

BM25 不擅长：

1. 语义近义表达
2. 口语化问法
3. 改写表达

例如：

```text
用户问：密码忘了怎么办
文档写：账号密码重置流程
```

词面不完全一致时，BM25 可能召回不稳。

### 2.8 BM25 和其他方案怎么选

#### 只用 BM25

优点：

1. 简单
2. 快
3. 对错误码、路径很稳

缺点：

1. 语义泛化弱

#### Elasticsearch / OpenSearch BM25

优点：

1. 更成熟
2. 支持更丰富查询能力

缺点：

1. 工程复杂度更高

#### 当前项目选择

项目把 BM25 当作“词面锚点召回器”，而不是唯一召回器。  
这就是最稳的用法。

### 2.9 可升级方向

1. 引入同义词词典
2. 增加 query expansion
3. 迁移到 Elasticsearch / OpenSearch
4. 结合字段权重检索

## 3. Dense 检索原理

### 3.1 Dense 检索解决什么问题

Dense 检索解决的是：

> 词不一定一样，但意思接近。

它最适合的场景是：

1. 用户问法口语化
2. 文档写法更规范
3. query 和文档存在同义改写

### 3.2 核心思想

Dense 检索先把：

1. query
2. document chunk

都编码到同一个向量空间里，然后通过向量相似度判断它们是否靠近。

本质上，它不直接比“词”，而是在比“语义位置”。

### 3.3 为什么向量空间能表达语义相似

embedding 模型训练时，会尽量把：

1. 语义相近文本拉近
2. 语义不相关文本拉远

所以类似下面这种情况，Dense 往往比 BM25 更稳：

```text
query: 密码忘了怎么办
doc: 账号密码重置流程
```

### 3.4 当前项目的实现

对应文件：

- [core/retrieval/dense_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/dense_retriever.py)

关键代码：

```python
self._get_model().encode(texts, normalize_embeddings=True)
sims = self._matrix @ q
```

### 3.5 为什么 `normalize_embeddings=True` 很重要

余弦相似度公式：

```text
cos(a, b) = a · b / (|a||b|)
```

当向量都归一化后：

```text
|a| ≈ 1, |b| ≈ 1
=> cos(a, b) ≈ a · b
```

这时直接做矩阵乘法就可以近似计算余弦相似度。

好处：

1. 算法简单
2. 批量计算高效
3. 实现清晰

### 3.6 当前项目为什么也是只索引 child chunk

和 BM25 一样，Dense 也只把 `searchable=True` 的 chunk 放进检索矩阵。

原因：

1. child 更适合召回
2. parent 太长时语义中心容易被摊薄
3. 检索阶段需要“敏捷命中”，不是“完整解释”

### 3.7 Dense 的优势

1. 处理语义近义表达
2. 处理 paraphrase
3. 对自然语言问法更鲁棒

### 3.8 Dense 的局限

1. 对错误码、路径、命令等精确 token 未必更强
2. 分数可解释性不如 BM25
3. 模型切换后，阈值和效果可能整体漂移

### 3.9 Dense 和其他向量方案怎么选

#### 本地矩阵 + SentenceTransformer

优点：

1. 简单
2. 好调试
3. 适合学习和中小规模数据

缺点：

1. 不适合超大规模

#### 向量数据库 + ANN

优点：

1. 更适合大规模召回
2. 查询更快

缺点：

1. 工程复杂度更高
2. 学习成本更高

### 3.10 可升级方案

1. 使用更强 embedding 模型
2. 接入 ANN 索引
3. 引入多向量检索
4. 按 query 类型选不同 embedding 模型

## 4. 混合召回与融合原理

### 4.1 为什么要混合召回

因为：

1. BM25 看词面
2. Dense 看语义

两者各有盲区。

如果只信其中一个：

1. 词面问题可能漏掉
2. 语义近义问题也可能漏掉

所以混合召回的本质不是“多搞一层复杂度”，而是：

> 同时覆盖两类相关性。

### 4.2 当前项目的融合入口

对应文件：

- [core/retrieval/hybrid_fusion.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/hybrid_fusion.py)

### 4.3 RRF 原理

RRF 全称 Reciprocal Rank Fusion。

它不比较原始分数，而是比较排名。

基础公式可以理解成：

```text
RRF score = Σ 1 / (k + rank_i)
```

假设某个 chunk：

1. 在 BM25 里排第 1
2. 在 Dense 里排第 3

那么：

```text
score = 1/(60+1) + 1/(60+3)
```

### 4.4 为什么 RRF 这么好用

因为不同检索器的原始分数通常不可比。

例如：

1. BM25 分数不是概率
2. 向量相似度也不是概率
3. 两者量纲完全不同

RRF 直接绕开这个问题，只看排名。

这意味着：

> 只要一个 chunk 在多路召回里都排得不错，它就会自然被抬起来。

### 4.5 Weighted Fusion 原理

Weighted Fusion 的思路是：

1. 先对不同召回器分数做归一化
2. 再做加权和

示意：

```text
final = sparse_weight * sparse_score + dense_weight * dense_score
```

它的优点是：

1. 可控性更强
2. 便于表达业务偏好

它的缺点是：

1. 需要调权重
2. 归一化方式会影响结果
3. 阈值更难解释

### 4.6 当前项目默认为什么更适合用 RRF

因为当前项目更强调：

1. 稳妥默认值
2. 教学可理解性
3. 少调参也能跑

所以 RRF 更适合做默认基线。

### 4.7 混合融合之后为什么还要门控

因为“融合出了结果”不等于“结果足够可靠”。

门控的本质是：

> 判断这些候选是否值得继续投入 rerank 和 generation 成本。

如果候选明显空或很弱，最稳的做法不是继续生成，而是尽早拒答。

### 4.8 可升级方案

1. 基于 query_type 动态切换融合策略
2. 加入学习排序
3. 引入文档新鲜度、来源可信度等特征
4. 扩展到多路召回融合

## 5. Cross-Encoder Rerank 原理

### 5.1 为什么召回后还要 rerank

召回阶段的目标是：

1. 别漏掉候选

重排阶段的目标是：

1. 从候选里挑出真正最值得给模型看的少量片段

所以二者目标不同：

```text
召回追求 recall
重排追求 precision
```

### 5.2 Cross-Encoder 的原理

Cross-Encoder 不像 Dense 那样分别编码 query 和 doc。  
它会把 query 和候选 chunk 一起送进模型。

也就是说，模型直接看的是：

```text
[query, chunk]
```

然后输出一个“这两者有多相关”的分数。

### 5.3 它为什么通常比简单相似度更准

因为它能建模更细粒度的交互关系：

1. query 里的某个词和 chunk 里的哪个句子对应
2. 两者是否真的在回答同一件事
3. 上下文里是否存在关键条件或否定关系

这也是为什么 Cross-Encoder 往往比 Dual-Encoder 更适合做精排。

### 5.4 当前实现

对应文件：

- [core/retrieval/reranker.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/reranker.py)

关键代码：

```python
pairs = [[query, c.content] for c in candidates]
scores = self._get_model().predict(pairs, show_progress_bar=False)
```

### 5.5 为什么它更准但更慢

因为它不是提前把所有文档向量算好再查，而是对每个候选都要做一次联合推理。

所以它适合：

1. 小规模候选精排

不适合：

1. 全库大规模召回

### 5.6 一个容易理解的例子

query：

```text
怎么处理 E-1001
```

两个候选：

```text
A: E-1001 常见原因与修复步骤
B: Redis 基础介绍与安装指南
```

召回阶段 A 和 B 都可能进来。  
但 Cross-Encoder 在联合看 `query + A`、`query + B` 后，通常会更稳地把 A 排前。

### 5.7 rerank 和 LLM 自己判断有什么区别

LLM 当然也能看上下文判断，但：

1. 直接把很多噪声上下文喂给 LLM 成本高
2. 也更容易被噪声诱导

rerank 的价值就在于：

> 在 generation 之前，先把候选压缩成更干净的证据集合。

### 5.8 可升级方案

1. 更轻量的 reranker
2. 分层 rerank
3. 服务化 rerank
4. 按 query 类型选择是否 rerank

## 6. Grounded Generation 原理

### 6.1 什么叫 grounded generation

Grounded generation 的意思是：

> 模型回答必须建立在当前提供的上下文之上，而不是凭常识自由发挥。

这也是 RAG 和普通聊天系统最核心的差别之一。

### 6.2 当前项目是怎么实现 grounded generation 的

对应文件：

- [core/generation/prompts/templates.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/generation/prompts/templates.py)
- [core/generation/context_format.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/generation/context_format.py)
- [core/orchestration/nodes/generate_answer.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/orchestration/nodes/generate_answer.py)

当前项目主要用 4 个手段约束模型：

1. 只喂重排后的上下文
2. prompt 明确要求只基于 CONTEXT 回答
3. 每段上下文都带 `CHUNK_ID`
4. 要求输出 `CITATIONS_JSON`

### 6.3 为什么 `CHUNK_ID` 很关键

因为 grounded 不只是“心理上相信模型有看上下文”，而是要能落到程序可校验。

`CHUNK_ID` 的价值：

1. 让模型引用时有稳定锚点
2. 后处理能做白名单校验
3. 前端能展示明确来源

### 6.4 为什么 `CITATIONS_JSON` 比纯文本引用更稳

如果只让模型输出：

```text
[CHUNK_ID:abc]
```

后处理只能靠正则去猜。  
而结构化 `CITATIONS_JSON` 更适合：

1. 稳定解析
2. 程序校验
3. 前端展示

### 6.5 当前方案的本质

本质上，当前项目是在把一个开放式生成问题收敛成：

```text
受限上下文
+ 可追溯引用
+ 结构化输出
```

这就是为什么它比“直接让模型回答”更接近生产级系统。

### 6.6 当前实现的局限

1. 仍然依赖模型遵守格式
2. 上下文过长时稳定性会下降
3. 目前引用粒度还是 chunk 级

### 6.7 可升级方案

1. JSON schema / function calling
2. 句子级引用
3. 答案-证据对齐评分
4. 更细的拒答模板

## 7. Grounding 校验原理

### 7.1 为什么模型输出后还要再校验

因为模型即使被 prompt 约束，仍然可能：

1. 漏引用
2. 引错引用
3. 编造引用
4. 低置信度胡说

所以 grounded generation 不等于 grounded answer。  
中间还必须有系统校验层。

### 7.2 当前项目的做法

对应文件：

- [core/orchestration/nodes/validate_grounding.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/orchestration/nodes/validate_grounding.py)

检查内容：

1. 非拒答答案是否带 citations
2. citations 是否属于当前上下文
3. confidence 是否低于系统阈值
4. 记录 citation coverage

### 7.3 这一层的工程意义

它把两件事彻底分开了：

1. 模型说了什么
2. 系统是否认可它

这就是企业系统和简单 Demo 的关键差别。

### 7.4 三种典型失败场景

#### 场景 A：没有引用

说明模型没有给出证据。  
系统通常应该下调置信度，必要时直接拒答。

#### 场景 B：引用不在当前上下文里

说明模型可能在编造引用。  
这种情况比“答错”更危险，因为它看起来像有证据。

#### 场景 C：置信度低于阈值

即使文本通顺，系统也可能拒答。  
这是在控制风险，而不是在追求“每次都要回答”。

### 7.5 为什么这是 RAG 的关键防线

如果没有这一层：

1. 低质量证据也能输出漂亮答案
2. 编造引用不容易被发现
3. 前端看到的会是“看起来像真的”结果

所以 grounding 校验不是装饰，而是：

> 模型输出进入用户界面之前的最后一道闸门。

### 7.6 可升级方案

1. 更细粒度证据校验
2. 句子级 grounding
3. 事实核验模型
4. 引入 answer-evidence alignment scorer

## 8. RAGAS 评测原理

### 8.1 为什么需要 RAGAS

因为 RAG 的质量不是一个单一维度问题。  
你至少要同时关心：

1. 答案是否忠实于证据
2. 答案是否答到点上
3. 是否召回了关键证据
4. 召回里是否混入太多噪声

### 8.2 当前项目的评测流程

对应文件：

- [core/evaluation/ragas_runner.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/evaluation/ragas_runner.py)

流程：

1. 读 JSONL 评测集
2. 对每个问题跑当前 RAG 系统
3. 收集 `answer` 和 `contexts`
4. 构造 HuggingFace `Dataset`
5. 调用 `ragas.evaluate`

### 8.3 当前使用的 4 个核心指标

#### Faithfulness

答案是否忠于上下文。

如果这个分数低，优先怀疑：

1. 幻觉
2. prompt 约束不够
3. grounding 失效

#### Answer Relevancy

答案是否真正回答了问题。

如果这个分数低，优先怀疑：

1. 回答跑偏
2. query 理解有问题
3. 生成组织方式不佳

#### Context Recall

系统是否召回了支持答案所需的关键证据。

如果这个分数低，优先怀疑：

1. 切块
2. query rewrite
3. 检索
4. fusion / rerank

#### Context Precision

召回上下文是否足够干净，而不是噪声很多。

如果这个分数低，优先怀疑：

1. top_k 太大
2. 检索太宽
3. rerank 不够准
4. chunk 太大

### 8.4 为什么不能只看 faithfulness

因为：

1. 一个答案可能很忠实，但没有真正回答问题
2. 一个答案也可能看起来相关，但证据召回其实不足

所以 RAG 评测应该永远是多指标联合看。

### 8.5 指标组合怎么帮助定位根因

#### `context_recall` 低 + `context_precision` 低

优先怀疑检索整体有问题。

#### `context_recall` 高 + `faithfulness` 低

优先怀疑证据到了，但模型没老实基于证据回答。

#### `context_recall` 高 + `faithfulness` 高 + `answer_relevancy` 低

优先怀疑答案虽然 grounded，但没真正答到点上。

### 8.6 当前实现为什么先串行执行

当前项目在教学环境里优先保证：

1. 日志更清楚
2. 错误更容易定位
3. 单题 badcase 更容易回溯

所以先串行比一开始就上高并发更适合学习。

### 8.7 可升级方案

1. 增加业务级评测集
2. 建立版本回归对比
3. 增加 badcase 分类体系
4. 建立人工评测闭环

## 9. 可观测性原理

### 9.1 为什么 RAG 项目更需要可观测性

因为 RAG 不是单一模型调用，而是多阶段系统：

1. API
2. 检索
3. 融合
4. 重排
5. 生成
6. 后处理

任何一层都可能：

1. 变慢
2. 出错
3. 质量下降

### 9.2 当前项目记录了哪些关键指标

对应文件：

6. `CITATION_COVERAGE`
7. `FAITHFULNESS_SCORE`

### 9.3 这些指标各自回答什么问题

#### `REQUEST_LATENCY`

整体接口是否变慢。

慢在 BM25、Dense 还是 Fusion。

#### `FAITHFULNESS_SCORE`

### 9.4 Logging 和 Tracing 的意义

对应文件：

- [core/observability/logging.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/observability/logging.py)

指标告诉你“哪里不对劲”。  
日志和 tracing 告诉你“到底是哪一步出了什么问题”。

这是两层不同的能力：

1. Metrics 适合看趋势
2. Logs / Traces 适合看单次问题

### 9.5 为什么可观测性本身也是质量能力

很多人把可观测性当成“附加功能”，其实在 RAG 里它本身就是质量体系的一部分。

因为如果你看不到：

1. 检索空召回率
2. rerank 延迟
3. token 消耗
4. 引用覆盖率

那你就很难知道问题到底在：

1. 数据
2. 检索
3. 生成
4. 基础设施

### 9.6 可升级方案

1. 增加 query_type 维度指标
2. 增加 badcase 采样日志
3. 把 tracing 和前端请求链路打通
4. 做版本级指标对比看板

## 10. 查询规划与 Milvus 后端原理

这一章专门讲你刚才指出的那条链路：

```text
策略选择模型 -> 多路查询 -> 向量检索后端 -> rerank -> answer
```

如果只讲 BM25、Dense、Fusion，而不讲“查询规划”和“Dense backend 分发”，你就会看到很多技术点，但看不清它们在真实问答链路里是怎么接起来的。

### 10.1 为什么不能只有一条 query

很多 RAG 教学资料会把检索写成：

```text
question -> rewrite -> retrieve
```

但生产场景里，一个问题往往同时包含多个检索切面。

例如：

```text
我有一个包含 100 亿条记录的数据集，想把它存到 Milvus 中进行查询，可以吗？
```

这个问题里同时混着：

1. 产品名：`Milvus`
2. 规模约束：`100 亿条记录`
3. 任务目标：`存进去并查询`
4. 判断诉求：`可不可以`

如果只保留一条 query，很容易只覆盖其中一个切面。

### 10.2 当前项目里的“策略选择模型”到底是什么

当前项目没有把它实现成一个名字就叫“策略选择模型”的类。

它实际上是由 3 层协作组成的：

1. `analyze_query_node`
   判断 query 类型，比如 `error_code / procedure / general`
2. `clarify_query_node`
   判断当前问题是否信息不足，需不需要先追问
3. `build_query_plan`
   生成多路查询计划

也就是说，当前项目把“策略选择”拆成：

```text
问题识别
  + 信息充分性判断
  + 多路查询规划
```

这样做的好处是：

1. 每一步职责更清楚
2. 出问题时更容易排查
3. 以后可以单独升级其中一层，而不用重写整块系统

### 10.3 `build_query_plan` 为什么是关键技术点

对应文件：

- [core/orchestration/query_expansion.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/orchestration/query_expansion.py)

它输出的不是一个字符串，而是一份结构化检索计划：

```python
QueryPlan(
    rewritten_query=...,
    multi_queries=[...],
    keyword_queries=[...],
    hyde_query="...",
    planning_summary="...",
)
```

这 5 个字段分别解决不同问题：

1. `rewritten_query`
   解决用户表达口语化、不适合检索的问题
2. `multi_queries`
   解决复杂问题只有一个检索视角不够的问题
3. `keyword_queries`
   解决错误码、术语、产品名需要精确词面命中的问题
4. `hyde_query`
   解决 query 太短、太抽象时 Dense 语义锚点不够的问题
5. `planning_summary`
   解决调试时“这次为什么这样搜”的可解释性问题

### 10.4 多路查询各自适合什么技术

#### `rewritten_query`

作用：

1. 把口语化问题收敛成更正式、更可检索的表达
2. 通常同时喂给 sparse 和 dense

#### `multi_queries`

作用：

1. 把一个复杂问题拆成多个检索切面
2. 扩大召回覆盖面

#### `keyword_queries`

作用：

1. 把高区分度 token 单独拿出来
2. 提高 BM25 的精确命中能力

适合场景：

1. 错误码
2. 路径
3. 产品名
4. 配置项
5. 缩写词

#### `hyde_query`

作用：

1. 先构造一个“可能的答案描述”
2. 再让 Dense 用这个描述去找语义接近的 chunk

### 10.5 HyDE 的原理到底是什么

HyDE 可以理解成：

```text
先写一段“可能长什么样的相关文档描述”，再把这段描述拿去做 embedding 检索
```

为什么这样可能有效：

1. 原问题有时太短
2. 原问题只有需求，没有知识表达
3. 文档库里的真正相关内容往往是陈述句、步骤句、说明句

举例：

```text
原问题：这个错误怎么处理
HyDE query：错误码排查通常需要先确认报错编号、环境、日志，并检查相关依赖服务状态
```

这样 Dense 更容易在语义空间里找到“故障排查类 chunk”。

它的风险也很明显：

1. 假设答案可能带偏
2. 如果模型幻觉很强，反而会污染召回方向

所以当前项目里 `hyde_query` 不是强制主路径，而是可选 route。

### 10.6 多路查询为什么最终要统一融合

因为不同 route 的结果分布不同：

1. `keyword_queries` 往往命中更精确，但覆盖面窄
2. `multi_queries` 往往覆盖面广，但噪声更多
3. `hyde_query` 可能召回语义相关但词面不一致的片段

如果直接“谁命中谁赢”，系统会非常不稳。

所以当前项目采用的是：

```text
route 内部先检索
  -> 所有 route 结果统一汇总
  -> child 粒度去重
  -> parent 回扩
  -> fusion
  -> rerank
```

本质上是在做：

> 先把不同路线都变成候选证据，再交给统一排序层决策。

### 10.7 为什么 Dense backend 要抽象成 file / milvus 两种实现

对应文件：

- [core/services/runtime.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/services/runtime.py)
- [core/retrieval/dense_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/dense_retriever.py)
- [core/retrieval/milvus_retriever.py](/Users/zhangzhijin/study/黑马学习/rag/RAG-demo1/enterprise-rag-platform/core/retrieval/milvus_retriever.py)

核心装配逻辑：

```python
if self.settings.vector_backend == "milvus":
    self._dense = MilvusDenseRetriever(self.settings)
else:
    self._dense = DenseRetriever(self.settings)
```

这里的设计思想非常重要：

1. 对编排层暴露统一接口
2. 把 backend 选择放到 runtime 装配层
3. 上层节点不关心底层向量是本地矩阵还是远端向量库

这是一种很典型的工程解耦思路：

```text
编排层依赖能力接口
运行时决定具体实现
```

### 10.8 为什么接了 Milvus 之后，本地 `IndexStore` 还要保留

当前项目不是：

```text
用了 Milvus 就把本地索引全部删掉
```

而是：

```text
本地 IndexStore 继续保留
  + Milvus 负责 dense retrieval
```

原因有 4 个：

1. BM25 仍然需要本地可读文本语料
2. parent 回扩仍然需要本地 chunk metadata
3. 调试时本地 `chunks.jsonl` 比远端 collection 更容易直接看
4. 服务重启时可以用本地快照补建 Milvus collection

这就是当前项目采用的工程折中：

```text
本地镜像负责“可读、可回查、可调试”
Milvus 负责“正式的 dense 向量召回”
```

### 10.9 如果做技术选型，这几种方案怎么比

#### 方案 A：只有本地 DenseRetriever

优点：

1. 简单
2. 本地调试方便
3. 学习成本低

缺点：

1. 数据规模上来后内存压力大
2. 检索能力和工程能力都有限

#### 方案 B：所有向量能力直接全切 Milvus

优点：

1. 更接近正式生产架构

缺点：

1. 调试成本更高
2. 需要额外解释 parent 回扩、BM25 语料来源、索引同步

#### 方案 C：当前项目方案，本地镜像 + 可切换 Dense backend

优点：

1. 学习友好
2. 工程过渡平滑
3. 对现有链路侵入小

缺点：

1. 需要维护两份语义上相关的数据视图
2. 需要关注本地快照和 Milvus collection 的一致性

### 10.10 以后还能怎么升级

1. 增加真正的 query router，为不同问题类型动态选择 route 集合
2. 让不同 route 使用不同 top_k、不同 fusion 权重
3. 对 `hyde_query` 做开关和质量监控，避免低质量 route 污染召回
4. 给 Milvus 增加 collection 分片、分租户或字段过滤能力
5. 把 route 级 trace 打到可观测性系统里，直接分析哪条 route 真正贡献了最终答案
