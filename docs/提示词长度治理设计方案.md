# 提示词长度治理设计方案

## 1. 结论

如果后续项目要正式支持多轮对话，又想把 API 成本稳住，最推荐的路线不是“简单截断历史消息”，而是把提示词治理升级成一套统一机制：

1. 引入 **统一 token 预算器**
2. 把历史对话拆成 **最近原文 + 滚动摘要 + 会话事实**
3. 对历史和证据都做 **语义重要性筛选**
4. 流式与非流式共用同一套 **结构化 prompt builder**

这样做的直接收益有 4 个：

1. 控制长对话下 prompt 无限膨胀
2. 让流式 / 非流式成本和行为保持一致
3. 在保留上下文连贯性的前提下显著降低 token 消耗
4. 为后续本地模型 / 外部模型混合路由保留统一接口

---

## 2. 当前项目为什么还不够

结合当前仓库实现，现状更接近“生成前瘦身”，还不是“完整的上下文窗口治理”：

1. 当前主回答主要通过 `generation_context_max_docs / max_chunks_per_doc / max_chars` 控制证据量
2. 非流式生成会做 context packing
3. 流式生成没有完全复用相同的 packing 逻辑
4. 历史消息当前主要用于 `resolve_context`，并没有正式进入最终回答 prompt
5. 长度控制主要基于 **字符数估算**，不是严格的 token 预算

这套机制对单轮 RAG 已经够用，但一旦未来支持真正的多轮对话，问题会很快暴露：

1. 会话越长，历史原文越容易挤占证据空间
2. 历史消息、治理提示、系统提示、检索证据会竞争同一个上下文窗口
3. 不同模型 tokenizer 差异会让“按字符数估算”越来越不稳
4. 如果直接拼接最近几轮原文，成本会持续上升
5. 如果直接粗暴丢弃历史，又会损失对话连贯性

所以，后续真正要上的不是一个“小优化”，而是一套分层治理方案。

---

## 3. 总体目标

后续方案需要同时满足这 5 个目标：

1. **保证连贯性**  
   用户上一轮确认过的实体、时间、部门、约束条件，不能轻易丢掉。

2. **保证任务完成质量**  
   当前问题最相关的检索证据必须优先保留，不能为了塞历史而牺牲 grounding。

3. **显著降低成本**  
   长会话下 prompt token 不能随着轮次线性增长。

4. **兼容企业治理约束**  
   `ACL / data classification / model_route / governance_notice` 仍然要进入统一 budget。

5. **兼容现有分层架构**  
   继续沿用 `settings -> orchestration -> generation -> api` 现有结构，不做推倒重来。

---

## 4. 推荐的总体方案

### 4.1 统一成一个 Prompt Budget Pipeline

建议把未来的提示词治理收敛成下面这条链路：

```text
question
-> session memory build
-> recent history scoring
-> evidence scoring / packing
-> token budget allocation
-> structured prompt build
-> LLM
```

核心思想是：

1. 不是先把所有东西拼起来，再看是否超长
2. 而是先按预算分区，再决定每个区块能保留多少内容

也就是说，后续的生成 prompt 应该按“预算驱动”来构造，而不是按“字段存在就拼接”来构造。

## 5. 具体该怎么做

这里按你当前项目的工程风格，给一套最稳的实施顺序。

### 5.1 第一步：引入统一 token 预算器

#### 目标

把当前分散的：

- `generation_context_max_chars`
- `max_tokens=1024`
- `history_messages[-4:]`
- `resolved_query[:512]`

升级成真正统一的预算视图。

#### 推荐做法

后续新增一个独立模块，例如：

```text
core/generation/prompt_budget.py
```

模块职责：

1. 计算模型可用输入预算
2. 给不同 prompt 区块分配 token 上限
3. 当预算不够时，按优先级逐步压缩
4. 输出统一的 `messages + token_usage_estimate + drop_reasons`

#### 推荐预算公式

```text
available_input_tokens
= context_window_tokens
- reserved_completion_tokens
- safety_margin_tokens
```

推荐默认值：

```text
context_window_tokens = 32768
reserved_completion_tokens = 1024
safety_margin_tokens = 512
```

#### 推荐区块划分

建议后续把生成 prompt 拆成 6 个区块：

1. `system_rules`
2. `current_task`
3. `active_session_memory`
4. `recent_relevant_turns`
5. `retrieved_evidence`
6. `governance_notice`

#### 推荐优先级

优先级建议如下：

1. `system_rules`
2. `current_task`
3. `retrieved_evidence`
4. `active_session_memory`
5. `governance_notice`
6. `recent_relevant_turns`

解释：

1. 企业 RAG 的本质是“当前任务 + 当前证据”优先
2. 历史上下文很重要，但不能反客为主
3. 最近几轮原文应该是最先压缩的部分，而不是证据块

---

### 5.2 第二步：把历史对话拆成三层记忆

如果未来真的支持多轮，最不推荐的做法就是：

```text
直接把最近 N 轮 user / assistant 原文全拼进 prompt
```

更推荐的做法是把会话记忆分成三层：

#### 第一层：`recent_exact_turns`

保留最近 2 到 4 轮最关键的原始消息。

适合保留的内容：

1. 最近一次用户追问
2. 最近一次 assistant 的关键回答
3. 刚确认过的约束条件
4. 当前未完成的任务说明

#### 第二层：`rolling_summary`

把更早历史压成结构化摘要，而不是保留原文。

建议摘要内容固定成：

1. 本会话主题
2. 已确认事实
3. 用户偏好或约束
4. 当前仍未解决的问题
5. 已经引用过的重要证据编号

不建议写成大段自由文本摘要，因为那样会：

1. 再次消耗大量 token
2. 难以稳定控制长度
3. 难以区分哪些事实必须保留

#### 第三层：`session_facts`

把关键事实单独存成结构化字段，例如：

```json
{
  "entities": ["安环部", "二矿", "安生平台"],
  "time_constraints": ["本月"],
  "user_constraints": ["必须给出处", "优先制度文件"],
  "unresolved_tasks": ["还未确认具体入口菜单"],
  "cited_chunk_ids": ["chunk-101", "chunk-208"]
}
```

这层的作用很大：

1. 结构化字段长度稳定
2. 对后续 query rewriting 和 retrieval 也有帮助
3. 不容易在压缩过程中丢掉关键锚点

---

### 5.3 第三步：做历史消息的智能压缩

#### 推荐压缩触发条件

满足任一条件就触发：

1. 会话轮数超过 `8`
2. 历史原文 token 超过 `recent_turns_max_tokens`
3. 当前问题显著依赖历史上下文
4. 历史区块挤占了证据预算

#### 推荐压缩策略

不要简单保留“最后 N 条消息”，而是分三步：

1. 先计算每条历史消息对当前问题的语义重要性
2. 只保留高分消息原文
3. 把其余消息压成 rolling summary

#### 历史消息评分建议

每条历史消息可按下面的综合分排序：

```text
history_score
= 0.35 * relevance_to_current_question
+ 0.20 * recency
+ 0.20 * entity_overlap
+ 0.15 * unresolved_task_bonus
+ 0.10 * explicit_constraint_bonus
- 0.15 * redundancy_penalty
```

优先保留的消息：

1. 用户明确给出的约束
2. 上一轮 assistant 的关键结论
3. 尚未完成的任务说明
4. 含部门、时间、系统名、设备名、文号的消息

最先压缩掉的消息：

1. 寒暄
2. 重复确认
3. 冗长解释
4. 与当前主题脱钩的旧消息

---

### 5.4 第四步：做证据块的语义重要性筛选

当前项目已经有一层 context packing，这很好，但后续要升级成“语义重要性 + 预算感知”的版本。

#### 证据选择推荐规则

每个 chunk 可以计算综合分：

```text
evidence_score
= 0.45 * retrieval_or_rerank_score
+ 0.20 * keyword_overlap
+ 0.15 * entity_overlap
+ 0.10 * authority_bonus
+ 0.10 * freshness_bonus
- 0.20 * redundancy_penalty
```

#### 推荐保留策略

1. 先保留分数最高的跨文档证据
2. 控制单文档 chunk 数，避免单文档垄断 prompt
3. 同一 `section_path` 只保留一个代表片段
4. 在预算不够时，优先对 chunk 做句子级压缩，而不是直接尾部截断

#### 为什么不能只做尾部截断

因为企业文档常见情况是：

1. 结论在前，条件在后
2. 规则在前，例外说明在后
3. 操作步骤是按编号分段的

如果直接裁末尾，很容易把：

- 适用条件
- 限制条款
- 例外说明

裁掉，导致回答看起来“像对的”，但其实不稳。

#### 推荐的句子级压缩方式

优先保留下面几类句子：

1. 包含实体名、版本号、部门名、文号的句子
2. 包含数值、阈值、时间范围的句子
3. 包含“必须 / 禁止 / 应当 / 不得 / 仅限 / 例外”的句子
4. 包含步骤动词的句子
5. 段首句和结论句

---

### 5.5 第五步：统一流式与非流式 prompt 构造

这一步非常关键。

未来正式落地时，流式和非流式一定要共用同一个：

```text
build_generation_messages(...)
```

原因：

1. 否则相同问题会走出两套 prompt
2. 两套 prompt 会带来长度、成本、答案风格和引用表现的差异
3. 后续调优时很难定位问题究竟来自“模式差异”还是“算法差异”

建议未来 builder 输出：

1. `messages`
2. `selected_history`
3. `packed_contexts`
4. `memory_summary`
5. `token_usage_estimate`
6. `drop_reasons`

这样：

1. API 层直接调 builder
2. Graph 非流式也直接调 builder
3. 审计日志可以记录同一口径的 preview 与预算使用

## 6. 多轮对话场景下的推荐结构

推荐后续真正送给模型的 prompt 长这样：

```text
SYSTEM_RULES:
- 只能基于证据回答
- 证据不足时拒答
- 必须返回 citation

CURRENT_TASK:
- question: ...
- resolved_query: ...
- answer_goal: ...

ACTIVE_SESSION_MEMORY:
- user_constraints: ...
- confirmed_facts: ...
- unresolved_tasks: ...

RECENT_RELEVANT_TURNS:
[USER] ...
[ASSISTANT] ...

RETRIEVED_EVIDENCE:
[CHUNK_ID:...] title=... source=... section=...
...

GOVERNANCE_NOTICE:
- conflict_summary: ...
- classification: ...

OUTPUT_SCHEMA:
ANSWER: ...
CONFIDENCE: ...
REASONING_SUMMARY: ...
CITATIONS_JSON: [...]
```

优点：

1. 区块边界清晰
2. 每个区块都能单独做预算控制
3. 更容易追踪到底是哪部分吃掉了 token
4. 更适合企业场景的 explainability 和审计

## 8. 推荐的落地顺序

为了最小改动、降低风险，建议分三期做。

### 第一期：统一预算和 prompt builder

目标：

1. 新增 token 预算器
2. 流式 / 非流式统一 prompt 构造
3. 把现在的 context packing 升级为 token-aware packing

这一期先不做复杂会话摘要，收益最大，风险最低。

### 第二期：引入 conversation memory

目标：

1. 最近原文保留
2. 更早历史滚动摘要
3. 关键会话事实结构化存储

这一期开始真正支持多轮，但仍然保持工程可控。

### 第三期：语义筛选和评测闭环

目标：

1. 历史语义打分
2. 证据句子级压缩
3. token 成本、答案质量、引用覆盖率联动评测

这一期做完，提示词治理才算从“能用”走到“可持续优化”。

## 10. 风险与注意事项

### 10.1 不要只做“最后 N 轮保留”

因为这会导致：

1. 关键约束可能在更早一轮
2. 最近几轮可能只是围绕细节反复确认
3. 会话一长，历史价值和时间顺序并不总是正相关

### 10.2 不要只做字符数裁剪

字符数不等于 token，中文场景尤其如此。

### 10.3 不要让历史压过证据

企业 RAG 的核心仍然是 grounded answer。  
历史上下文再重要，也不能挤掉高价值检索证据。

### 10.4 摘要必须结构化

如果 rolling summary 变成大段自由文本，很容易：

1. 再次膨胀
2. 丢失关键字段
3. 难以稳定控制预算

## 11. 推荐的第一轮实施范围

如果后续真的开始写项目代码，我最推荐第一轮只做这几件事：

1. 引入统一 token 预算器
2. 统一流式 / 非流式 prompt builder
3. context packing 从 `max_chars` 升级为 token-aware
4. 先接最近原文历史，不接复杂 rolling summary

原因：

1. 收益最大
2. 风险最低
3. 最容易验证
4. 不会一下子改太多核心链路

## 15. 按阶段拆任务时怎么拆最合适

如果后续你要让我或别的开发同学正式落代码，我建议按下面方式拆 task。

### 15.1 第一轮 task

目标：

1. 新增 `prompt_budget.py`
2. 新增 token 统计和 budget 数据结构
3. `generate_answer.py` 改成走统一 builder
4. `chat.py` 流式路径复用同一 builder

这一轮不要碰会话摘要。

### 15.2 第二轮 task

目标：

1. 新增 `conversation_memory.py`
2. `resolve_context.py` 补 `session_facts`
3. `state.py` 增加 memory 字段
4. prompt builder 开始接最近原文历史

这一轮只接“最近原文 + 结构化 facts”，先不做复杂摘要模型。

### 15.3 第三轮 task

目标：

1. 增加 rolling summary
2. 增加历史语义评分
3. 增加 evidence 句子级压缩
4. 增加 debug / metrics / audit 补充字段

这样拆的优点：

1. 每轮目标单一
2. 不会一次改太多核心文件
3. 每轮都能独立验证收益

## 17. 一句话落地建议

如果后续真的开始接代码，最稳的做法是：

> 先把“流式 / 非流式统一 prompt builder + token 预算器”做好，再逐步把多轮 memory 接进来；不要一开始就把会话摘要、历史筛选、证据压缩、调试字段全部一起上。

原因很简单：

1. 第一阶段收益最大
2. 第一阶段最容易验证
3. 第一阶段对当前单轮 RAG 风险最小
4. 做完第一阶段，后面再加多轮 memory 才不会乱
