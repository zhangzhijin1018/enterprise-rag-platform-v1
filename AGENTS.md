# AGENTS.md

## 默认工作方式

- 默认使用中文回答；代码、命令、路径、报错和日志保留原文。
- 默认按生产级工程场景理解问题，优先给可落地、可维护、适合长期迭代的方案。
- 回答时遵循：先给结论，再给步骤，最后给示例。
- 信息不足时先做合理假设，并明确说明假设。
- 修改代码时优先最小改动，不擅自改变接口、目录结构、数据格式和返回值约定，除非用户明确要求。

## 项目定位

### 项目名称

新疆能源（集团）有限责任公司 企业知识智能副驾

### 一句话定义

这是一个基于现有 `enterprise-rag-platform` 演进的企业级 RAG 系统，面向新疆能源集团内部制度、技术文档、运维手册、项目资料、会议纪要等多源知识场景，重点提供可追溯、可控权、可评测、可审计的知识问答与辅助决策能力。

### 核心目标

1. 提升企业内部知识查询效率，降低查制度、查流程、查技术资料的时间成本。
2. 让回答基于证据、可引用、可追溯，减少幻觉和不可靠输出。
3. 在检索阶段落实权限控制和数据分级，避免越权访问。
4. 建立评测闭环、审计日志和可观测能力，支撑持续优化。
5. 保持现有工程化骨架，在不推倒重来的前提下完成企业化定制。

### 典型知识源

- 集团制度文件
- 安全生产规范
- 设备运维手册与 SOP
- 项目设计文档
- 招采与合同流程文件
- 会议纪要
- 故障复盘与应急预案
- API / 平台技术文档

### 典型用户

- 普通员工
- 部门负责人
- 运维工程师
- 项目经理
- 信息化 / AI 团队
- 管理层

## 产品原则

1. 不是通用聊天机器人，而是企业知识智能副驾。
2. 不是只回答问题，而是必须基于证据回答。
3. 不是“拿到文档就都能看”，而是必须做权限过滤。
4. 不是“有内容就生成”，而是证据不足时必须拒答。
5. 不是 demo 导向，而是生产级工程导向。
6. 不是默认把所有内容送外部模型，而是遵循最小必要出域原则。
7. 所有新增能力优先复用现有架构，不做无关重构，不推翻既有模块分层。

## 当前仓库现状认知

在开始改动前，默认基于以下事实理解当前项目：

1. 当前仓库已经具备企业级 Agentic RAG 骨架，包含 ingestion、retrieval、orchestration、generation、evaluation、observability、deployment 等分层。
2. 现有主链路基于 FastAPI + LangGraph + hybrid retrieval + rerank + grounded generation，不允许随意替换主框架。
3. `core/config/settings.py` 是统一配置中心；新增配置默认放这里。
4. `apps/api/routes/chat.py` 是在线问答核心入口；新增请求上下文、安全策略和响应字段优先从这里接入。
5. retrieval 层已支持 metadata/filter 透传，后续 ACL、数据分级、权威性和版本策略优先沿这条链路扩展。
6. 当前任务是“企业场景定制化和安全化改造”，不是从零重建新项目。

## 长期架构边界

### 必须保留的架构分层

- `ingestion`
- `retrieval`
- `orchestration`
- `generation`
- `evaluation`
- `observability`
- `deployment`

### 优先沿用的技术路线

- FastAPI API 层
- LangGraph 编排层
- hybrid retrieval（dense + sparse + metadata filter）
- rerank 二阶段检索
- grounded answer + citation
- RAGAS / 自定义评测
- Prometheus / OpenTelemetry

### 不允许的方向

- 推倒现有目录结构重新设计
- 为了“代码更优雅”做大规模无关重构
- 引入与当前主架构冲突的全新框架
- 把项目改造成通用闲聊机器人

## 企业化改造的强制要求

### 1. 权限控制

- 权限必须在检索前或检索中完成，禁止生成后再裁剪敏感内容。
- chunk 入库时应逐步补齐 ACL / 部门 / 角色 / 项目组 / 文档密级等 metadata。
- 在线问答必须支持携带用户身份上下文，如 `user_id`、`department`、`role`、`project_ids`、`clearance_level`。
- 无权限访问时要明确返回无权限或证据不可用，不得隐式降级为“编造答案”。

### 2. 数据分级与模型路由

- 支持至少四类数据级别：`public`、`internal`、`sensitive`、`restricted`。
- 高敏数据默认遵循最小必要出域原则。
- `restricted` 数据默认不出域，只允许本地模型、摘要模式或拒答。
- 模型路由、是否允许外部 LLM、是否需要脱敏，都应通过配置和上下文显式控制。

### 3. 引用溯源

- 回答尽量返回引用，不允许无依据直接生成结论。
- 引用信息优先包含：文档名、`chunk_id`、章节/页码、版本、生效日期、权威级别。
- 流式和非流式模式的引用字段应尽量保持一致。

### 4. 拒答机制

- 证据不足、重排分过低、上下文冲突严重或权限不足时，应优先拒答。
- 拒答要返回明确原因，例如：证据不足、无权限、数据不可出域、知识库无相关内容。
- 禁止为了“看起来更智能”而在低证据场景下自由脑补。

### 5. 冲突检测

- 多文档结论冲突时，不允许把冲突隐藏成单一模糊答案。
- 应尽量输出 `conflict_detected`、`conflict_summary` 或同等语义字段。
- 冲突处理可基于版本优先、制度优先、权威来源优先等策略，但必须显式说明依据。

### 6. 审计与日志

- 关键问答链路应逐步补齐审计信息：用户身份、查询、检索命中文档/Chunk、模型路由、是否拒答、置信度等。
- 日志中禁止记录敏感全文；必要时只记录摘要、哈希、文档 ID、Chunk ID 等最小必要信息。
- 审计和日志能力必须与现有 observability 体系兼容。

## 代码与实现规则

1. 默认优先输出 Python，其次是 bash、SQL、YAML。
2. 代码尽量可直接运行，不输出只停留在概念层的伪代码。
3. Python 代码尽量包含必要类型注解、简洁 docstring、必要异常处理和清晰变量命名。
4. 默认优先最小侵入式改造，不做过度封装、过度设计或无关重构。
5. 新增配置统一放入 `core/config/settings.py`。
6. 新增 API 字段必须同步更新 schema、route、tests、docs。
7. 新增 retrieval / generation / orchestration 能力时，优先复用现有接口和状态结构。
8. 重要改动必须补单元测试；影响接口行为时补最小集成测试。
9. 不得伪造测试结果；无法执行的测试要明确说明原因。
10. 文档、示例配置和 mock 数据的更新要与代码改动同步，而不是事后补。
11. 每完成一轮代码改动，必须同步补齐对应文档。
12. 同步补文档时，至少检查并按需更新以下文档层次：
    - 对外说明：`README.md`、`docs/api.md`、`docs/architecture.md`
    - 学习型文档：`功能链路详解.md`、`核心技术原理.md`、`项目架构文档.md`、`源码精读手册.md`、`源码阅读路线图.md`
13. 文档更新不能只写“变更摘要”，要把当前版本的真实链路、设计取舍、影响范围、限制和后续方向写清楚。

## 任务执行要求

每次开始改代码前，优先先输出：

1. 对目标的理解
2. 本轮修改的文件列表
3. 每个文件修改原因
4. 风险点
5. 验证方法

未明确上述内容前，不要直接做大范围改动。

### 推荐的实施顺序

1. `settings + schema`
2. `chat route + runtime context`
3. `retrieval ACL / data classification filter`
4. `ingestion metadata 增强`
5. `generation refusal / conflict`
6. `audit logging`
7. `evaluation / tests / docs`

### 单轮改动约束

- 一次只完成一个明确目标。
- 不要一次修改超过 5 个核心文件，除非用户明确要求。
- 每轮改动后给出变更摘要、验证步骤、影响范围、风险和未完成项。
- 每轮改动后默认把相关代码文档一起补齐；如果本轮不补，必须明确说明为什么。
- 如果发现当前工作区已有用户修改，优先兼容，不要直接覆盖或回滚。

## 面试导向要求

所有方案和代码改造，都应尽量服务于以下面试表达：

1. 这是一个真实企业场景，不是通用 demo。
2. 项目重点是“答得准、答得稳、能引用、可控权限、可评测、可持续优化”。
3. 优先体现检索质量、权限控制、拒答机制、引用溯源、评测闭环和工程化能力。
4. 避免只堆砌“agent”概念；复杂能力应建立在可靠 RAG 主链路之上。

## 禁止事项

1. 禁止把权限控制做成生成后裁剪。
2. 禁止默认把敏感上下文完整发送给外部模型。
3. 禁止只改 schema 不改调用链、测试和文档。
4. 禁止为了方便引入不必要的重型依赖。
5. 禁止输出与现有项目阶段不匹配的理想化空架子代码。
6. 禁止无依据承诺“已验证通过”或“可直接上线”。
