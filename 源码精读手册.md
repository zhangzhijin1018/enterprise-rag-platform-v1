## 1. 怎么使用这份手册

这份文档不是项目概览，也不是算法原理文档。  
它的目标非常明确：

> 带你按文件读源码，并告诉你每个文件最值得抓住的点。

每个文件我都尽量按同一套模板写：

1. 文件作用
2. 核心输入输出
3. 关键函数 / 类
4. 调用链位置
5. 阅读顺序
6. 你读完后应该能回答什么问题

最推荐的使用方式是：

1. 先看这里的“阅读重点”
2. 再打开对应源码

## 3. 问答主链路精读

这一章你可以先抓一条最容易迷路、但也最值得抓住的真实调用链：

```text
chat.py
  -> graph.py
  -> analyze_query / clarify_query / resolve_context
  -> rewrite_query.py
  -> query_expansion.py
  -> retrieve_docs.py
  -> runtime.dense
  -> DenseRetriever 或 MilvusDenseRetriever
  -> hybrid_fusion.py
  -> reranker.py
  -> generate_answer.py
```

你刚才问的“策略选择模型 -> 多路查询 -> Milvus”其实就在这条链上，只是以前文档没有把它单独拎出来。

### 3.0 当前版本新增的企业化阅读提示

如果你现在读这个项目的源码，不能再只按“普通 RAG”来理解。

最近两轮之后，问答主链路已经多了这些企业化语义：

1. `ChatRequest` 不再只是一句问题，而是“问题 + 企业用户上下文”
2. `/chat` 入口会先构造 `user_context / access_filters / audit_id`
3. 带企业安全上下文的请求会有条件绕过尚未 ACL 化的 fast path
4. `retrieve_docs_node` 已经开始同时消费 `structured_filters + access_filters`
5. `RAGState` 里已经增加了 `data_classification / model_route / answer_mode / audit_id`
6. 检索结果在 child hits 阶段会做 ACL fallback，而不是只靠底层检索器
7. 最近又新增了 `risk_level / risk_action / risk_reason / risk_require_alert`
8. 这轮又新增了更厚的企业 metadata schema，以及 chunk 级 `section_path / section_type / topic_keywords`

所以你读源码时，建议脑子里同时挂着两条线：

```text
传统 RAG 线：query -> retrieve -> rerank -> generate
企业安全线：user_context -> access_filters -> ACL -> classification -> route tag
```
