# Enterprise RAG Platform - 源码阅读路线图

## 使用方式

这份路线图不是“泛泛建议”，而是给你按顺序学这个项目用的。

建议做法：

1. 每次只读一小组文件。
2. 每读完一组，都实际跑一个请求或看一个数据文件。
3. 先建立主链路，再回头补细节。

如果你每天能投入 1 到 2 小时，这份路线图大概可以按 7 天完成第一轮。

### 当前版本新增的学习提醒

最近两轮之后，这个项目已经不只是“普通 RAG 骨架”，而是开始进入“企业知识智能副驾”的演进阶段。

所以你看源码时要额外关注这条企业安全线：

```text
ChatRequest
  -> user_context
  -> access_filters
  -> retrieve(filters = structured + access)
  -> ACL fallback
  -> data_classification
  -> model_route
  -> ChatResponse / NDJSON
```

如果你忽略这条线，很容易只看懂“怎么答”，却看不懂“为什么现在要这么改”。

另外最近入库链路也有一个很值得你重点看的变化：

```text
不同 parser 先增强结构
  -> 统一 SemanticChunker
  -> 按文件类型选择 chunk profile
```

这意味着你后面读 ingestion 时，不要再把它理解成“所有文档都按同一把尺子硬切”。

### 你要回答的问题

- `/chat` 为什么要区分流式和非流式？

- `top_k` 为什么不直接等于最初召回数？
- `history_messages` 为什么要在入口就透传到后续链路？
- `user_context / access_filters / audit_id` 为什么在路由层就要构造？
- 为什么当前版本对带企业安全上下文的请求会绕过 fast path？

### 配套操作

非流式：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "错误码 E-1001 是什么？应如何处理？",
    "top_k": 8,
    "stream": false
  }'
```

带历史消息的多轮请求：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "今天是谁值班",
    "history_messages": [
      {"role": "user", "content": "冲压二车间夜班安排表在哪"},
      {"role": "assistant", "content": "请看共享盘排班目录"}
    ],
    "top_k": 8,
    "stream": false
  }'
```

带企业安全上下文的请求：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "生产环境数据库扩容 SOP 是什么",
    "department": "信息化部",
    "role": "engineer",
    "project_ids": ["proj-a"],
    "clearance_level": "internal",
    "allow_external_llm": false,
    "top_k": 8,
    "stream": false
  }'
```

这条请求最值得观察的是：

1. 响应里已经开始带 `audit_id`
2. 会返回 `model_route / data_classification / refusal_reason`
3. 当前版本如果识别到企业安全上下文，会优先绕过尚未 ACL 化的 fast path

流式：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "错误码 E-1001 是什么？应如何处理？",
    "top_k": 8,
    "stream": true
  }'
```

## 学习时的三个建议

### 1. 不要一开始就追求看完所有代码

先把主链路打通比什么都重要。

### 2. 每读一个模块，都问自己“输入、输出、依赖、调用方是谁”

这是比背概念更有效的阅读方式。

### 3. 边读边跑

RAG 项目如果只看不跑，很容易停留在“知道有这个模块”，但不会真正理解模块之间如何协作。
