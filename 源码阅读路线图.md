# Enterprise RAG Platform - 源码阅读路线图

## 使用方式

这份路线图不是“泛泛建议”，而是给你按顺序学这个项目用的。

建议做法：

1. 每次只读一小组文件。
2. 每读完一组，都实际跑一个请求或看一个数据文件。
3. 先建立主链路，再回头补细节。

如果你每天能投入 1 到 2 小时，这份路线图大概可以按 7 天完成第一轮。

## 第 1 天：先把项目跑起来，建立整体感觉

### 目标

- 知道项目有哪些目录

- 知道系统“长什么样”

2. `pyproject.toml`
3. `Makefile`
4. `docker-compose.yml`
5. `apps/api/main.py`

### 你要回答的问题

- 这个项目是做什么的？
- API 怎么启动？
- 前端怎么连接后端？
- 运行时依赖有哪些？

### 配套操作


### 你要重点看的函数

- `_run_graph`
- `get_runtime_dep`
- `get_runtime`
- `reload_index`

### 你要回答的问题

- `/chat` 为什么要区分流式和非流式？
- 为什么路由层不直接 new 检索器和模型？
- `top_k` 为什么不直接等于最初召回数？

### 配套操作

非流式：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "错误码 E-1001 是什么？应如何处理？",
    "top_k": 8,
    "stream": false
  }'
```

流式：

```bash
curl -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{
    "question": "错误码 E-1001 是什么？应如何处理？",
    "top_k": 8,
    "stream": true
  }'
```

## 学习时的三个建议

### 1. 不要一开始就追求看完所有代码

先把主链路打通比什么都重要。

### 2. 每读一个模块，都问自己“输入、输出、依赖、调用方是谁”

这是比背概念更有效的阅读方式。

### 3. 边读边跑

RAG 项目如果只看不跑，很容易停留在“知道有这个模块”，但不会真正理解模块之间如何协作。

## 学完这一轮后你应该具备的能力

如果你把这份路线图走完，应该至少能做到：

1. 自己解释一次 `/chat` 的完整执行流程
2. 自己解释一次文件入库流程
3. 知道 chunk、embedding、BM25、Dense、Fusion、Rerank 在系统里的具体位置
4. 知道当前项目为什么适合学习、为什么还不能直接当大规模生产方案
5. 能自己继续扩展一个模块，比如新增 parser、加一个检索策略或替换索引层
